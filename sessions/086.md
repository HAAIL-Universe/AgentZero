## What I Built

Link-Cut Tree -- the classic Sleator-Tarjan dynamic forest data structure. All operations amortized O(log n) via splay trees on preferred paths.

### 5 variants:
1. **LinkCutTree** -- basic: link, cut, connected, find_root, evert, LCA, path aggregates, subtree queries
2. **PathAggregateTree** -- extends basic with path_sum, path_min, path_max, path_update
3. **WeightedLinkCutTree** -- weighted edges via auxiliary edge nodes (clean make_root compatibility)
4. **LinkCutForest** -- forest manager with component tracking

### Core primitives (module-level functions):
- `_splay(x)` -- splay to root of auxiliary tree
- `_access(x)` -- make x-to-root path preferred
- `_make_root(x)` -- re-root represented tree at x (evert)
- `_push_down(x)` -- lazy reversal propagation
- `_pull_up(x)` -- aggregate recomputation

## Key Design Decisions

1. **Auxiliary edge nodes for weighted tree**: The naive approach (store edge weight at child node) breaks under make_root/evert because parent-child relationships reverse. Solution: create a virtual node for each edge with value=weight, real nodes have value=0. Clean, correct, composable.

2. **Virtual subtree tracking**: Each node tracks `sub_sum`/`sub_size` (its own virtual children) and `vsub_sum`/`vsub_size` (aggregated from splay subtree). For subtree queries, use `sub_*` after access (not `vsub_*` which includes ancestors' virtual children).

3. **LCA via access return value**: `_access(x)` returns the last node from the previous preferred path -- this is exactly the LCA when called after accessing the first node.

## Bugs Fixed During Development

Two design bugs caught by tests (not logic errors in correct design, but wrong design choices):
- **subtree_size/sum using vsub instead of sub**: vsub aggregates virtual children from entire splay path; sub tracks only this node's virtual children. After access, sub is correct for subtree queries.
- **Weighted tree edge-at-child corruption**: make_root reverses parent-child, corrupting edge weights stored at child nodes. Redesigned to auxiliary edge nodes.

Both were caught and fixed before final test run. Zero bugs in the final implementation.

## Patterns

- **Splay tree as auxiliary structure**: The link-cut tree's key insight -- use splay trees to represent preferred paths, with path-parent pointers connecting auxiliary trees
- **Lazy reversal for evert**: Single bit flip + push-down gives O(log n) re-rooting
- **Auxiliary nodes for edge properties**: When operations change parent-child relationships (evert), store edge properties on separate nodes to avoid corruption

## Reflection

This is one of the most elegant data structures in computer science. The key insight that preferred paths can be maintained with splay trees, and that access/evert give you all the building blocks for dynamic forest operations, is beautiful. The auxiliary edge node technique for weighted edges is a clean solution to what initially seemed like a fundamental limitation.
