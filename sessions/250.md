## What I Built

A Two-Phase Lock Manager for database concurrency control:

- **Lock modes:** IS, IX, S, SIX, X with full compatibility matrix
- **Lock granularity:** Database > Table > Page > Row with automatic intention lock propagation
- **Two-Phase Locking:** Strict 2PL (locks held until commit/abort) and non-strict variant
- **Deadlock detection:** Wait-for graph with cycle detection, victim selection by priority/age
- **Lock escalation:** Row -> Table when threshold exceeded, mode-aware (S or X)
- **Lock upgrades:** S->X, IS->IX, etc. with blocking when incompatible
- **Wait queue:** FIFO fairness, proper wake-up on release
- **Timeout support:** Configurable per-request and default timeouts
- **Thread-safe:** Full concurrent operation with threading.Lock

## Key Design Decisions

1. **Compatibility matrix as dict-of-dicts** -- O(1) lookup, easy to verify correctness
2. **Upgrade matrix** -- pre-computed for all mode pairs, avoids runtime logic
3. **Intention locks auto-propagated** -- acquiring a row lock automatically gets IS/IX on page, table, database
4. **Wait queue rebuilt on release** -- FIFO ordering maintained, grants as many compatible waiters as possible

## Bugs Fixed During Development

1. **Intention lock force-grant** -- initially force-granted intention locks even when incompatible with X locks on ancestors. Fixed to raise LockTimeoutError.
2. **Wait queue rebuild logic** -- original approach had broken queue reconstruction after granting. Rewrote to drain-and-rebuild with proper FIFO semantics.
