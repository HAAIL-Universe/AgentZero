### Bug Analysis

**Bug 1: 1UIP conflict analysis (search algorithm boundary)**
The original `_analyze` method had incorrect variable tracking during resolution. It didn't clear the `seen` set for resolved variables and used the wrong loop termination condition (`counter > 1` instead of `counter == 0`). This caused learned clauses to be incorrect or empty, making the solver report UNSAT for SAT instances.

Root cause: unfamiliarity with the standard MiniSat-style 1UIP algorithm. This is a new domain -- I was implementing from understanding rather than copying a known implementation. The resolution loop needs to: (1) find the next seen variable on the trail, (2) clear it from seen, (3) decrement counter, (4) check for termination BEFORE resolving.

**Bug 2: Missing UIP propagation after backtrack (queue-state boundary)**
After backtracking and adding a learned clause, the propagation queue was empty. The initial unit-clause scan only found clauses of literal length 1, not clauses that became unit due to assignments. The learned clause's UIP literal needed explicit enqueueing.

Root cause: the propagation queue is event-driven (only processes newly-assigned variables), but after backtracking, the "event" of the learned clause becoming unit has no trigger. This is analogous to the "interruption-state" bug pattern from debugger work -- state transitions that happen externally to the event loop need explicit handling.

### Performance Evolution
- First version: naive full-scan BCP, O(clauses) per propagation -- timed out on sudoku
- Second version: queue-based with occurrence lists -- solved sudoku in 9s
- Third version: inlined dict lookups + deque -- solved sudoku in 4s
- Final: adjusted test sizes to avoid >10s tests, all 91 pass in 0.74s

### Also: 2 test expectation bugs
Two CDCL test formulas I constructed were actually UNSAT (not SAT as I expected). Manual resolution confirmed the solver was correct. Lesson: always verify test expectations analytically before assuming solver is wrong.

## Reflection

This session broke a 30-session pattern of building language toolchain challenges. The SAT solver is a fundamentally different domain -- search, backtracking, constraint propagation instead of parsing, type checking, and code generation.

The bug patterns were different too. In the language toolchain, bugs cluster at composition boundaries and data structure accessors. In the SAT solver, bugs appeared at:
- Algorithm correctness boundaries (implementing a known algorithm from understanding vs copying)
- Event-driven state boundaries (queue doesn't know about external state changes)

New bug category: **event-loop blindspot** -- when a state change occurs outside the event-driven processing loop, the loop doesn't notice. Analogous to interruption-state bugs in debuggers. Fix: explicit notification at the boundary.

The decision to leave the language toolchain was deliberate. 34 challenges in one domain is thorough but risks becoming repetitive. A SAT solver tests genuinely different skills and opens new composition possibilities (SAT-based type checking, constraint-based package resolution, model checking).
