## Root Cause Analysis

The bug affected ALL expression-statements (Assign, IndexAssign, DotAssign, CallExpr) when used inside function/method bodies. These node types leave a value on the VM stack:

- **Assign**: emits value + DUP + STORE (net +1 on stack)
- **IndexAssign/DotAssign**: emits obj + index + value + INDEX_SET (INDEX_SET pops 3, pushes 1 = net +1)
- **CallExpr**: emits callee + args + CALL (CALL pops args+callee, pushes result = net +1)

When these appear as **statements** (not sub-expressions), the leaked value is never consumed. The values accumulate on the shared stack. When RETURN pops only the return value but not the leaked values, the caller's stack frame is corrupted -- the outer CALL picks up leaked values instead of the intended function reference.

**Example**: `assertEqual(s.pop(), 2)` where `pop()` contains `this.size = this.size - 1;` (a DotAssign statement). The DotAssign leaks `size-1` onto the stack. After RETURN, the stack has `[assertEqual_fn, leaked_value, return_value]` instead of `[assertEqual_fn, return_value]`. The outer `CALL 1` picks up `leaked_value` as the "function" to call.

## The Fix

Added `_compile_stmt(node)` method to the Compiler that emits POP after expression-statement nodes. Applied in three places:
1. `compile_Block` -- all block bodies (if, while, for, try, class methods)
2. `_compile_function_body` -- function body statements
3. `compile` (top-level) -- all statements except the last (last value = program result)

Statement types (no POP needed): LetDecl, LetDestructure, Block, IfStmt, WhileStmt, FnDecl, ReturnStmt, PrintStmt, ForInStmt, ForInDestructure, BreakStmt, ContinueStmt, TryCatchStmt, ThrowStmt, ImportStmt, ExportFnDecl, ExportLetDecl, ExportClassDecl, ExportEnumDecl, ClassDecl, EnumDecl, AsyncFnDecl, ExportAsyncFnDecl.

Everything else is an expression that needs POP when used as a statement.

## Why This Bug Survived So Long

1. Most test code uses `let` declarations (which don't leak) rather than reassignment
2. Reassignment inside functions is rare in the test patterns -- most tests use temp variables
3. The bug only manifests in NESTED calls (inner function result as argument to outer function)
4. Session 053 partially masked it by adding stack cleanup in constructors
5. Top-level expression-statements don't cause issues (no outer call frame to corrupt)
