# Session 310 -- Database Query Optimization

**Date:** 2026-03-18
**Focus:** Agent Zero database performance -- query consolidation, pagination, composite indexes

## What I Did

### 1. Collapsed get_productivity_summary() from 7 queries to 3
- **Before:** 7 sequential DB round-trips (4 fetch_val for event counts, 1 fetch_val for active count, 2 fetch_all for streaks/cadence)
- **After:** 3 queries using PostgreSQL FILTER clause for conditional aggregation
  - Query 1: `fetch_one` with 4x FILTER clauses for all event counts in one scan
  - Query 2: `fetch_all` with GROUP BY cadence (active count derived from sum)
  - Query 3: `fetch_all` for streaks
- **Impact:** At 50 concurrent users, 350 QPS -> 150 QPS (57% reduction in DB round-trips)

### 2. Added LIMIT/OFFSET pagination to list_requests()
- `update_requests.py`: Added `limit` (default 100, max 500) and `offset` (default 0) parameters
- Input validation: clamp limit to [1, 500], offset to [0, inf)
- `agent_zero_server.py`: Updated `/api/requests` endpoint to accept `limit` and `offset` query params with FastAPI validation
- **Impact:** Fixes OWASP API4:2023 violation (unbounded result sets)

### 3. Added 4 composite indexes to SCHEMA_SQL
- `idx_commitments_user_status ON commitments(user_id, status)` -- used 3x in productivity
- `idx_commitment_events_user_type_date ON commitment_events(user_id, event_type, occurred_at DESC)` -- used 4x
- `idx_goals_user_created ON goals(user_id, created_at DESC)` -- for sorted goal listing
- `idx_commitment_events_user_date ON commitment_events(user_id, occurred_at DESC)` -- for event history
- All use `CREATE INDEX IF NOT EXISTS` for idempotent migration

## Key Design Decisions

- **3 queries instead of 2:** The paper proposed collapsing to 2 using window functions + jsonb_agg. I chose 3 for readability -- the cadence GROUP BY naturally gives active count as sum, and streaks are a separate simple query. The critical optimization (4 event queries -> 1 via FILTER) is preserved.
- **Active count from cadence sum:** Instead of a separate COUNT query, `active_commitments` is now `sum(by_cadence.values())`. Same result, zero extra queries.
