## What I Built

**C100: Piece Table** -- the text editing data structure used by VS Code.

Instead of modifying text in place, it maintains two buffers:
- **Original buffer** (immutable) -- the initial text
- **Add buffer** (append-only) -- all inserted text

A table of "pieces" references slices of these buffers. Edits modify only the piece table, never the text buffers. This makes undo/redo trivial and memory-efficient.

### Features
- Two-buffer architecture (original + add, append-only)
- O(pieces) insert/delete with piece splitting
- Full undo/redo via command history
- Line index with offset-to-line/col conversion
- Boyer-Moore-Horspool string search
- Find-and-replace (single and all occurrences)
- Marks (bookmarks/anchors with gravity that track through edits)
- Multiple cursors with selection support
- Snapshot/versioning for state restoration
- Compact (merge adjacent contiguous pieces)
- Line-level operations (insert_lines, delete_lines, replace_line)
- Character iteration

### Key Design Decisions
- line_starts stored per-piece as relative offsets for efficient line counting
- Boundary line_starts (at split points) stay with left piece (`ls <= po`) since the \n that causes them is in the left piece
- Delete right-remainder uses strict `ls > end_po` to avoid counting deleted newlines
- Marks have gravity (left/right) controlling behavior at insertion point
- Undo reverses mark positions alongside piece table state

## Bug Found and Fixed During Development
- **Line_starts boundary splitting** -- when splitting a piece at position `po`, a line_start at exactly `po` (caused by \n at `po-1`) must be assigned to the LEFT piece, not the right. The \n is in the left piece, so the line boundary marker stays with it. Fixed by using `ls <= po` for left and `ls > po` for right in both insert and delete operations.
