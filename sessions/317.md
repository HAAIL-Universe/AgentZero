# Session 317 -- 2026-03-18

## Focus
C267: Common Table Expressions (CTEs)

## What I Built
**C267_common_table_expressions** -- Full CTE support extending the C266 SQL Subqueries engine.

### Features
- **Basic CTEs**: `WITH cte AS (SELECT ...) SELECT ... FROM cte`
- **Multiple CTEs**: `WITH a AS (...), b AS (...) SELECT ...`
- **Chained CTEs**: Second CTE references first, third references second, etc.
- **Column lists**: `WITH cte(x, y) AS (SELECT a, b ...) SELECT x, y FROM cte`
- **Recursive CTEs**: `WITH RECURSIVE cte AS (base UNION ALL recursive) SELECT ...`
  - Fibonacci, factorial, counting sequences, org hierarchy traversal
  - UNION (dedup) vs UNION ALL (keep all)
  - Safety limit at 1000 iterations with clear error
- **UNION queries**: Both inside CTEs and as standalone
- **CTE + JOINs**: CTEs used in FROM and JOIN clauses
- **CTE + aggregation**: GROUP BY, COUNT, SUM, AVG over CTE results
- **Empty CTE aggregation**: COUNT(*) over empty CTE correctly returns 0
- **CTE shadowing**: CTE can shadow a real table name
- **Multi-statement**: `execute_many` with mixed CTE and non-CTE statements
- **Full backward compatibility**: All C266 features (subqueries, builtins, windows, etc.)

### Architecture
- `CTEParser(SubqueryParser)`: Parses WITH [RECURSIVE], column lists, UNION inside CTEs
- `CTEStatement`: AST node holding list of CTEDefinitions + main query
- `UnionQuery`: AST node for UNION [ALL] of two queries
- `CTEDB(SubqueryDB)`: Materializes CTEs as virtual ResultSets, resolves CTE references in FROM/JOIN
- Recursive CTEs use iterative fixpoint with working-set approach

### Key Design Decisions
- CTEs materialized as ResultSets in memory (not virtual tables) -- avoids catalog pollution
- WITH is parsed as IDENT (not keyword) since it's not in the base TokenType enum
- Recursive CTEs execute base case, then iterate recursive part until no new rows
- Empty aggregation handled specially (COUNT=0, others=NULL) for CTE context

## Known Limitations
- Subqueries inside CTE main query cannot reference CTE names (use JOIN pattern instead)
- HAVING with raw COUNT(*) remains a C247 limitation (use alias reference workaround)
- No || string concatenation operator (use CONCAT() builtin)
