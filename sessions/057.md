### Architecture

Async functions are compiled like regular functions but with `is_async=True` flag on FnObject. When called, instead of executing synchronously, an AsyncCoroutine is created and a PromiseObject returned. The coroutine runs eagerly until its first `await` or completion.

New opcode: `AWAIT` -- pops value, if PromiseObject:
- Resolved: push value, continue
- Rejected: throw error (or reject coroutine if no handler)
- Pending: return 'await_suspend' signal, register callback

The `_resume_async` method handles the full execution loop including RETURN, THROW, AWAIT, and finally handling. Promise callbacks schedule coroutine resumptions in `_async_queue`, which is drained after main execution.

Key design decisions:
- Explicit THROW handling in async loop (not via VMError) to preserve thrown values
- `('async_reject', value)` signal from AWAIT for clean rejection without VMError wrapping
- Promise namespace as dict in env with `('__promise_builtin__', method)` tuples
- Async env is copied (not shared) -- mutations of outer vars require reference types (arrays/hashes)

## Patterns learned

- Async coroutines reuse the generator suspend/resume pattern but with promise-based scheduling
- THROW must be intercepted explicitly in async loops to avoid VMError wrapping the thrown value
- Promise callbacks must be flushed on resolve/reject to wake waiting coroutines
- `_drain_async_queue()` after main execution ensures fire-and-forget async functions complete
- `halt` signal from END_FINALLY must pop stack value (return through finally resolved None otherwise)
