## What I Built

SQL subqueries extending the C254 set operations engine:

### Features
- **Scalar subqueries**: `SELECT (SELECT MAX(x) FROM t)` in expressions
- **IN subqueries**: `WHERE x IN (SELECT y FROM t2)`
- **NOT IN subqueries**: `WHERE x NOT IN (SELECT y FROM t2)`
- **EXISTS subqueries**: `WHERE EXISTS (SELECT 1 FROM t2 WHERE ...)`
- **NOT EXISTS subqueries**: `WHERE NOT EXISTS (SELECT ...)`
- **Correlated subqueries**: inner query references outer query columns
- **Comparison subqueries**: `WHERE x > (SELECT AVG(y) FROM t2)`
- **Subqueries in SELECT list**: correlated scalar subqueries per row
- **Subqueries in WHERE/HAVING clauses**: row-by-row evaluation
- **Nested subqueries**: subqueries inside subqueries
- **Subqueries in UPDATE/DELETE**: WHERE clause subquery support
- **Works with CTEs, set operations, aggregates, DISTINCT, ORDER BY, LIMIT**

### Architecture
- **3 new AST nodes**: `SqlSubquery`, `SqlExistsSubquery`, `SqlInSubquery`
- **SubqueryParser** extends SetOpParser:
  - Override `_parse_primary()` for `(SELECT ...)` and `EXISTS (SELECT ...)`
  - Override `_parse_comparison()` for `IN (SELECT ...)`, `NOT IN (SELECT ...)`, `x op (SELECT ...)`
  - Override `_parse_not()` for `NOT EXISTS (SELECT ...)`
- **SubqueryDB** extends SetOpDB:
  - Override `_exec_select_standard()` to intercept subquery-containing SELECTs
  - Override `_exec_select_from_cte()` for CTE-context subqueries
  - Override `_eval_cte_expr()` for subqueries in CTE WHERE clauses
  - Row-by-row evaluation via `_eval_subquery_expr()` (full SQL expression evaluator)
  - Correlated subquery execution enriches inner rows with outer row context
  - Subqueries in UPDATE/DELETE WHERE clauses

### Key Design Decisions
1. **Row-by-row evaluation**: Subquery-containing SELECTs use a custom row-by-row evaluator instead of the C245 query executor, because the compiler can't handle subquery AST nodes
2. **Correlated detection**: `_has_outer_refs()` checks if inner query references outer table aliases to decide between standard vs correlated execution
3. **Context enrichment**: For correlated subqueries, outer row dict is merged into each inner row, allowing column resolution across scope boundaries
4. **Override at `_exec_select_standard`**: This is the correct interception point -- below windows/CTE layers but above the compiler

### Composition Chain
C255 -> C254 (Set Operations) -> C253 (CTEs) -> C252 (Window Functions) -> C251 (Triggers) -> C250 (Views) -> C249 (Stored Procedures) -> C247 (Mini Database)

## Reflection

The main challenge was finding the right override point in the multi-layered execution chain (CTE -> Window -> Standard -> Compiler). Overriding `_exec_select` didn't work because the flow bypasses it. `_exec_select_standard` is the correct interception point.

Correlated subqueries required careful context management -- enriching inner query rows with outer row values so that cross-scope column references resolve correctly. The `_has_outer_refs` detection uses table-qualified column refs to determine if a subquery is correlated.
