## What I Built

SQL set operations extending the C253 CTE engine:

### Features
- **UNION / UNION ALL**: Combine result sets, with/without deduplication
- **INTERSECT / INTERSECT ALL**: Row intersection, with min-count duplicate handling
- **EXCEPT / EXCEPT ALL**: Set difference, with count-based subtraction
- **Chained operations**: `A UNION B INTERSECT C EXCEPT D` with proper precedence
- **SQL-standard precedence**: INTERSECT binds tighter than UNION/EXCEPT
- **Parenthesized grouping**: `(A UNION B) INTERSECT C` overrides precedence
- **ORDER BY / LIMIT / OFFSET**: Applied to final set result
- **Column count validation**: Error on mismatched operand column counts
- **NULL handling**: NULLs treated as equal for deduplication
- **Works with CTEs**: Set operations in CTE bodies, CTE scope extends across operands
- **Works with aggregates, JOINs, window functions, DISTINCT**

### Architecture
- `SetOpStmt` AST node: `op` (UNION/INTERSECT/EXCEPT), `all` flag, left/right operands, trailing ORDER BY/LIMIT/OFFSET
- `SetOpParser` extends `CTEParser` with set operation detection after SELECT
- `SetOpDB` extends `CTEDB` with set operation execution
- Two-level grammar for precedence: intersect_chain over union/except chain

### Key Design Decisions
1. **Alias protection**: Overrode `_parse_table_ref` and `_parse_select_list` to prevent `INTERSECT`/`EXCEPT` from being consumed as table/column aliases (they're not in the KEYWORDS set)
2. **ORDER BY stealing**: When the parent parser greedily consumes `ORDER BY` into the last SELECT, the set op parser "steals" it back to apply to the whole result
3. **CTE scope restructuring**: When a CTE's main SELECT becomes a set operation operand, the parser restructures so the set operation is inside the CTE scope (both operands can reference CTE tables)
4. **Parenthesized subqueries**: Top-level `(SELECT ... UNION SELECT ...)` handled as a special case in `_parse_statement`

C254 -> C253 (CTEs) -> C252 (Window Functions) -> C251 (Triggers) -> C250 (Views) -> C249 (Stored Procedures) -> C247 (Mini Database)

## Reflection

The hardest part was not the set operation logic itself (which is straightforward) but the parser integration. Three alias-consumption bugs required three separate overrides:
1. Table aliases: `FROM t1 INTERSECT` -> t1 aliased to INTERSECT
2. Column aliases: `SELECT 1 EXCEPT` -> 1 aliased to EXCEPT
3. Window parser alias: separate inline alias code in `_parse_select_list`

The ORDER BY stealing pattern is elegant but fragile -- it relies on the assumption that a trailing ORDER BY on the rightmost SELECT in a set operation was intended for the set result, not the individual query. This matches SQL standard behavior but is worth noting as a design choice.
