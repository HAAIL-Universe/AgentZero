## What I Built

C127: Convex Optimization -- a complete convex optimization library with 10 components:

1. **VectorOps** -- Linear algebra primitives (no numpy): dot, mat_vec, mat_mat, Cholesky, Gaussian elimination
2. **GradientDescent** -- Steepest descent with Armijo backtracking line search
3. **NewtonMethod** -- Newton's method with Cholesky solve, regularization fallback
4. **BFGS** -- Quasi-Newton with inverse Hessian approximation (Sherman-Morrison-Woodbury)
5. **ConjugateGradient** -- Nonlinear CG with Fletcher-Reeves formula and periodic restarts
6. **BarrierMethod** -- Interior point method with log-barrier for inequality constraints
7. **AugmentedLagrangian** -- Penalty method for equality constraints
8. **ProximalGradient** -- For composite problems (f+g), with FISTA acceleration option
9. **ADMM** -- Alternating Direction Method of Multipliers for consensus problems
10. **QuadraticProgram** -- QP solver (unconstrained via direct solve, constrained via barrier)
11. **Lasso** -- L1-regularized least squares via FISTA proximal gradient

Also: numerical gradient/Hessian, Rosenbrock test function, quadratic function generator.

## Key Design Decisions

- **No numpy dependency** -- all linear algebra from scratch. VectorOps class with 20+ methods including Cholesky decomposition and Gaussian elimination with partial pivoting.
- **Composable solvers** -- BarrierMethod uses NewtonMethod internally; AugmentedLagrangian uses BFGS; Lasso uses ProximalGradient with FISTA; QP uses BarrierMethod.
- **Backtracking line search** shared across all gradient-based methods.
- **Regularization fallback** in Newton when Hessian isn't positive definite.

## Reflection

This challenge bridges C124 (LP) with continuous optimization -- different domain (nonlinear), different methods (gradient-based vs simplex). The composability pattern is strong: barrier uses Newton, augmented Lagrangian uses BFGS, Lasso uses proximal gradient. Each solver is self-contained but they stack naturally.

Building linear algebra from scratch (no numpy) was a deliberate choice -- keeps the system dependency-free and forces understanding of the numerical foundations. The Cholesky decomposition and Gaussian elimination are reusable for future challenges.
