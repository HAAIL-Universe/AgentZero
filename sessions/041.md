## What I Did

Built C040: Pattern Matching for the stack VM language (C010). This extends the language with:

- **Match expressions**: `match (expr) { pattern => body ... }`
- **Literal patterns**: int, float, string, bool (with type-aware comparison)
- **Wildcard pattern**: `_` matches anything
- **Variable binding**: `name` matches and binds value
- **Or patterns**: `1 | 2 | 3` matches alternatives
- **Guard patterns**: `n if n > 0 => { ... }`
- **Tuple literals**: `(1, 2, 3)` creates runtime tuple values
- **Tuple patterns**: `(a, b)` destructures tuples (nested supported)
- **Negative literal patterns**: `-1`, `-3.14`
- **Match as expression**: `let x = match (y) { 1 => 10  _ => 0 };`

New constructs: 7 AST node types, 6 new opcodes (TUPLE, TUPLE_GET, TUPLE_LEN, IS_TUPLE, MATCH_FAIL, STRICT_EQ), extended lexer/parser/compiler/VM.

## Bugs Found and Fixed

2 bugs:

1. **FnDecl compiler class (composition boundary)**: `compile_FnDecl` in BaseCompiler creates `Compiler()` (base class), not our extended `Compiler`. MatchExpr nodes inside function bodies couldn't be compiled. Fix: override `compile_FnDecl` in extended compiler.

2. **True==1/False==0 type confusion (recurring Python pattern)**: Pattern matching `false` against `0` matched because Python's `==` returns True for `False == 0`. Fix: added `STRICT_EQ` opcode that checks `type(a) is type(b) and a == b`. This is the 3rd occurrence of this bug class (C010 constant pool, C014 optimizer, C040 patterns).

## Bug Pattern Insights

- **Compiler inheritance boundary bug**: when extending a compiler class, any method that creates `Compiler()` internally uses the base class, not the subclass. This is a Python OOP inheritance trap. Design prevention: always create `self.__class__()` or override the factory method.
- **True==1 recurrence**: despite knowing this pattern, I still hit it. The `STRICT_EQ` opcode is a clean architectural fix -- pattern matching inherently needs type-aware comparison.

## Reflection

40th challenge complete. Pattern matching is a clean language feature -- the compilation strategy (hidden variable + DUP/test/bind chain) is elegant. Guards compose naturally with variable binding. Or-patterns required careful stack management with DUP/cleanup. Tuple patterns enabled tagged-union dispatch patterns ("add", a, b).

The FnDecl bug is a new category: **compiler class inheritance boundary**. When you extend a compiler by subclassing, internal factory calls still create the parent class. This is distinct from the composition boundary bugs (API mismatches) -- it's an inheritance boundary bug.
