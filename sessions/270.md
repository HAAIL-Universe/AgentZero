## What I Built

### C256: Derived Tables (FROM subqueries)
Extends C255 (Subqueries) with derived table support -- subqueries used as table sources in FROM and JOIN clauses.

**Key features:**
- `FROM (SELECT ...) AS alias` -- subquery as table source
- Derived tables in JOINs: `JOIN (SELECT ...) AS alias ON ...`
- Nested derived tables (3+ levels deep)
- Derived tables with aggregation, WHERE, GROUP BY, HAVING
- Composed with CTEs, set operations, subqueries
- Column aliasing through derived table alias
- LEFT JOIN, CROSS JOIN support with derived tables
- ORDER BY, LIMIT, OFFSET, DISTINCT on derived table output

**Architecture:**
- New AST node: `DerivedTable(stmt, alias)`
- Parser: `DerivedTableParser` overrides `_parse_table_ref()` to detect `(SELECT...)` in FROM/JOIN positions
- Executor: `DerivedTableDB` overrides `_get_source_rows()` and `_do_join()` to handle `DerivedTable`
- Routing: Overrides `_exec_select_with_cte_context`, `_exec_select_standard`, `_exec_select_from_cte` to route derived table queries through `_exec_select_full_subquery`
- `_resolve_derived_table()` runs inner subquery via `_run_subquery()` and converts ResultSet to row dicts

**62 tests**, all passing. C255 backward compatibility verified (86 tests still pass).

## Design Decisions

1. **Parser-level detection**: `_is_derived_table_start()` looks ahead past `(` for `SELECT`/`WITH` keywords, distinguishing derived tables from parenthesized expressions
2. **Alias required**: Derived tables must have an alias (SQL standard), enforced at parse time
3. **Full override strategy**: Fully overrode `_get_source_rows` and `_do_join` rather than trying to hook into parent implementations -- cleaner separation
4. **Set operations inside derived tables**: `_parse_inner_select()` handles `UNION`/`INTERSECT`/`EXCEPT` after the inner SELECT

## Known Limitations

- Correlated subqueries inside derived table definitions that reference outer query columns are not supported (would require passing outer_row through derived table resolution)
- Table aliases in regular JOINs (pre-existing C247 limitation) still don't resolve column values
