## What I Built

### C258: B-Tree Indexed Database
Composes C116 (B+ Tree) with C247 (Mini Database) to add real B+ tree indexes to the SQL engine. Previously, C247 stored index metadata but never used it -- all queries were full table scans. Now indexes are backed by actual B+ tree data structures and the query optimizer uses them.

**Architecture:**

1. **IndexManager** -- Manages B+ tree indexes (BPlusTreeMap per index). Handles create/drop, maintains indexes on INSERT/UPDATE/DELETE, supports equality and range lookups. Composite indexes use tuple keys. NULL values excluded from indexes.

2. **QueryOptimizer** -- Rule-based optimizer that analyzes WHERE clauses. Detects equality (`col = val`), range (`col > val`, etc.), and AND conditions (extracts indexable part, keeps remainder as post-filter).

3. **IndexedDB** -- Extends MiniDB with index-aware DML (index maintenance) and index-aware SELECT (optimizer decides seq vs index scan).

**Features tested:**
- Index creation with auto-population from existing data
- CREATE UNIQUE INDEX with violation detection
- DROP INDEX with fallback to seq scan
- Index maintenance on INSERT, UPDATE, DELETE
- All comparison operators via index
- AND conditions with partial index coverage
- ORDER BY, LIMIT, DISTINCT, aggregates with index scans
- EXPLAIN shows IndexScan vs SeqScan
- Multiple indexes per table, text columns, NULL handling
- Stress test: 1000 rows, correctness verification (index = seq results)
- Full backward compatibility with C247

**91 tests**, all passing. C257 (63 tests) backward compatibility verified.

## Design Decisions

1. **BPlusTreeMap: value -> [rowid list]**: Non-unique indexes map key to rowid list. Unique indexes enforce single-entry.
2. **Single-table optimizer only**: JOINs fall back to seq scan. Join index optimization needs a cost model.
3. **SqlLogic.operands[]**: C247 models AND/OR with operands list, not left/right. Fixed during development.
