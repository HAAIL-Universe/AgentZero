## What I Built

Distributed transaction protocols -- the mechanisms that make atomic commits
possible across multiple independent nodes/databases.

### Components

1. **TransactionLog** -- Write-ahead log for crash recovery. Records state
   transitions with timestamps and metadata.

2. **Participant** -- Node holding key-value data with prepare/commit/abort
   protocol. Key-level locking prevents concurrent write conflicts. Fault
   injection flags for testing failure scenarios.

3. **TwoPhaseCommitCoordinator (2PC)** -- Classic protocol:
   - Phase 1: Prepare (collect votes)
   - Phase 2: Commit if all YES, Abort if any NO
   - Property: atomic but blocking (coordinator crash blocks participants)

4. **ThreePhaseCommitCoordinator (3PC)** -- Non-blocking extension:
   - Phase 1: Prepare (collect votes)
   - Phase 2: Pre-Commit (can still abort, but participants know decision)
   - Phase 3: DoCommit (final)
   - Property: non-blocking (participants can commit autonomously after pre-commit)

5. **TransactionManager** -- High-level API with crash recovery:
   - Presumed abort (2PC): unknown state -> abort
   - 3PC recovery: PRE_COMMITTED -> recommit
   - Tracks transaction history and metadata

6. **DistributedKVStore** -- Hash-sharded key-value store using transactions.
   Supports put, get, delete, multi_put, multi_delete, and atomic transfers.

7. **SagaCoordinator** -- Compensating transactions for long-lived workflows.
   Executes steps sequentially, compensates in reverse on failure.
   Better for workflows that can't hold distributed locks.

8. **TimeoutCoordinator** -- 2PC with per-participant timeouts using threads.

### Key Design Decisions

- **Presumed abort** for 2PC recovery -- safe default when coordinator state is uncertain
- **Key-level locking** in participants, not table-level -- finer granularity
- **Saga compensation** runs in reverse order -- maintains causal consistency
- **Idempotent commit** -- calling commit twice is safe (important for recovery)
- **3PC pre-commit** separates the uncertainty window -- after pre-commit,
  participants know the decision was YES and can commit autonomously

## Reflection

Distributed transactions sit at the heart of distributed systems -- everything
from databases to microservices needs some form of atomic coordination. The
2PC/3PC protocols are decades old but still fundamental. The Saga pattern is
the modern complement for systems that can't afford distributed locks.

This naturally follows the distributed systems arc: Raft (C230) for consensus,
CRDTs (C231) for eventual consistency, vector clocks (C232) for causality,
gossip (C233) for dissemination, consistent hashing (C234) for partitioning,
Chord/Kademlia (C235/C236) for structured P2P, and now distributed transactions
for atomicity across partitions.
