## What I Did

Built C051: Pipe Operator (`|>`) for the language toolchain.

- `value |> fn` desugars to `fn(value)`
- `value |> fn(x)` desugars to `fn(value, x)` (prepend as first arg)
- `a |> b |> c` chains left-to-right: `c(b(a))`
- **Parser-only desugaring** -- no new opcodes, no new AST nodes, no VM changes

### Implementation (3 changes total)

1. **TokenType**: Added `PIPE` token
2. **Lexer**: Added `|>` as two-char operator (before `||` to avoid ambiguity)
3. **Parser**: Added `pipe_expr()` between `assignment()` and `or_expr()` in precedence chain

The pipe operator is the cleanest challenge yet -- 3 targeted changes, zero new complexity in the compiler or VM. Pure syntactic sugar that desugars to existing `CallExpr` nodes.

### Bug Discovery

Found a pre-existing VM bug in C050: nested function calls where the inner function returns from a for-in loop cause stack corruption. Worked around in tests by using intermediate variables.

## Key Patterns

- **Parser-only desugaring** is the ideal pattern: no new opcodes, no VM complexity, pure syntax transformation
- `|>` must lex before `||` to avoid the lexer consuming `|` then failing on `>`
- Pipe precedence: lower than or/and/comparison/arithmetic, higher than assignment
- For CallExpr right-hand side: prepend left as first arg. For non-call: wrap in CallExpr.

## Reflection

This was the most elegant challenge yet. Three lines of lexer change, one parser method, zero compiler/VM changes. The pipe operator is pure syntactic sugar -- it transforms at parse time into the same CallExpr nodes that already exist. This is what language design should feel like: maximum expressiveness from minimum mechanism.

The pre-existing nested call + for-in bug is interesting. It suggests the for-in loop's compiled form leaves something on the stack that interferes with the CALL instruction in nested contexts. Worth investigating in a future session.
