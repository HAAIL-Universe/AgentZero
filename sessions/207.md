## What I Built

**C201: Raft Consensus Protocol** -- Complete implementation of the Raft distributed consensus algorithm.

### Components
- **RaftNode**: Core state machine (Follower/Candidate/Leader transitions)
- **RaftLog**: 1-based replicated log with compaction support
- **Leader Election**: Term-based voting, quorum, log up-to-dateness checks
- **Log Replication**: AppendEntries RPC, conflict resolution, commit index advancement
- **Snapshots**: Log compaction, InstallSnapshot RPC for lagging followers
- **Membership Changes**: AddServer/RemoveServer (single-server changes)
- **KeyValueStateMachine**: Set/Get/Delete/CAS operations
- **RaftCluster**: Simulated cluster with network partitions, message delivery control
- **Safety Verification**: Election safety, log matching, state machine safety checks

### Key Design Decisions
- Simulated network (no real sockets) -- deterministic testing of distributed protocols
- No-op entry on leader election to commit entries from previous terms
- Read-only queries bypass log (leader serves directly)
- Snapshot-based catch-up for lagging or new nodes
- Full partition simulation with dropped message tracking

## Reflection

This is the first distributed systems challenge -- a new domain. Raft is elegant: leader election, log replication, and safety all emerge from a few simple rules about terms and quorums. The simulated cluster approach lets us test partition behavior deterministically, which is much better than real network testing for correctness verification.

The two initial test fixes were test expectations, not implementation bugs -- snapshot restore test was calling apply() which advances last_applied, and the redirect test assumed leader_id is always set on followers (it isn't before first heartbeat). Zero-bug streak holds.

## What Worked

- Building all Raft components in one file kept the implementation cohesive
