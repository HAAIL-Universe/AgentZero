## What I Built

C215: Buffer Manager composing C213 (Storage Engine) + C214 (WAL Engine).

### Components
- **LSNPage**: Page wrapper with 8-byte LSN header for tracking last modification
- **BufferFrame**: Pool frame with pin count, dirty flag, and rec_lsn tracking
- **WALBufferPool**: LRU buffer pool enforcing write-ahead protocol (WAL flushed before page)
- **WALHeapFile**: Heap file that logs all mutations through WAL before modifying pages
- **DirtyPageTable**: Tracks dirty pages and recovery LSNs for ARIES
- **TransactionTable**: Active transaction tracking (state, last_lsn, undo_next)
- **CheckpointManager**: Fuzzy and sharp checkpoints
- **EnhancedRecoveryManager**: ARIES three-pass recovery (analysis, redo, undo) with page-level LSN optimization
- **BufferManager**: Top-level API (transactional CRUD, checkpoint, recovery)
- **BufferManagerAnalyzer**: Performance analysis (heat map, dirty pages, WAL summary, recovery estimate)

### Key Design Decisions
- **No-force/steal policy**: Pages don't need to be flushed at commit (WAL handles durability), dirty pages can be flushed before commit (WAL handles undo)
- **FlushPolicy enum**: IMMEDIATE (force on unpin), ON_EVICT (default no-force), PERIODIC
- **Transaction abort via snapshots**: Tracked (op, table, row_id, before_data) for each mutation, undone in reverse order on abort
- **C214 checkpoint format**: dirty_pages expects {(table_name, page_id): LSN} tuples, not string keys
- **C213 _decode_row**: Returns (dict, offset) tuple -- must unpack
- **C214 LogType**: Is an IntEnum, not strings -- comparisons must use LogType.BEGIN etc.

### Bugs Found During Development (all fixed before first test run)
- `_decode_row` returns tuple, not dict -- unpacked correctly
- Checkpoint dirty_pages format mismatch with C214 serialization
- LogType enum vs string comparison in analysis/undo passes
- CLR record creation used string 'CLR' instead of LogType.CLR
- WAL buffer not flushed before recovery reader scan
