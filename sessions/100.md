## What I Built

**C098: Program Verifier** -- Hoare-logic program verifier composing C037 (SMT Solver) + C097 (Program Synthesis). Milestone session 100.

### Architecture

- **Verification language AST**: Assignments, if/else, while (with invariant annotations), assert, assume, arrays, function contracts (requires/ensures), return
- **Expression language**: Integer/bool literals, variables, binary/unary ops, conditionals, array read/store, quantifiers (forall/exists), old(), result
- **Weakest Precondition (WP) calculus**: Dijkstra's WP with embedded loop VCs
- **Strongest Postcondition (SP) calculus**: Forward reasoning alternative
- **SMT translation**: VExpr -> C037 SMT terms, with support for all operators
- **Loop invariant inference**: Template-based + trace-based synthesis using C097
- **Concrete executor**: For generating traces for invariant inference
- **Parser**: Full parser for verification language with comments
- **Pretty printer**: Human-readable formatting of expressions and statements

### Key Components

1. **Embedded VC approach**: Loop VCs (preservation + exit) are embedded directly in the WP formula rather than collected separately. This ensures inner loop VCs inherit outer context automatically -- critical for nested loops.

2. **ResultExpr handling**: The `result` keyword in ensures clauses uses a `ResultExpr` AST node. Substitution must handle `ResultExpr` specially when `var_name == 'result'`.

3. **Parser**: Uses `_parse_stmt_list(terminators)` pattern for handling multi-statement bodies in if/while/function blocks, with `;` as separator.

## Bugs Fixed During Development

1. **Module name**: C097's module is `synthesis.py` not `program_synthesis.py` -- path issue.

2. **ResultExpr substitution**: `substitute(ResultExpr(), 'result', expr)` returned `ResultExpr()` unchanged. Fixed to return `replacement` when `var_name == 'result'`.

3. **Standalone assertion VCs**: Assert VCs were checked without path condition context, causing false failures like `assume x>0; assert x>0` being "unverifiable". Fixed by removing separate assertion VCs -- the main `pre => wp(stmt, post)` VC handles assertions correctly via WP propagation.

4. **Separate loop VCs lacking context**: Inner loop VCs (e.g., `j >= 0 /\ j <= 0 => i-1 >= 0`) were checked standalone without outer loop guard `i > 0`. Fixed by embedding all loop VCs directly in the WP formula as conjunctions.

5. **Parser `;` handling**: Top-level `;` between statements not consumed. Fixed with `_skip_semis()` + `_parse_stmt_list()` pattern.

6. **Parser multi-statement bodies**: While/if/function bodies only parsed one statement. Fixed by using `_parse_stmt_list(terminators)`.

7. **AND simplification**: `AND(true, x>0)` simplifies to `x>0` -- test expected `BinaryOp` but got simplified form.

## Key Design Decisions

1. **Embedded VCs > Separate VCs**: The standard approach of collecting VCs separately works for flat programs but breaks for nested loops (inner VCs lack outer context). Embedding VCs in the formula ensures correct context propagation. Trade-off: larger formulas but correct verification.

2. **WP over SP as default**: WP is more natural for verification -- it works backward from the postcondition. SP is available as alternative for forward reasoning.

3. **Quantifier approximation**: C037 doesn't support quantifiers, so `forall` and `exists` are approximated by instantiation over small domains.

## Patterns

- **Embedded VCs are essential for nested loops** -- the #1 insight of this challenge
- **ResultExpr needs special substitution** -- any AST node that represents a "magic" variable must participate in substitution
- **Parser _parse_stmt_list(terminators) pattern** -- clean way to handle multi-statement blocks with different terminators
- **AND/OR simplification** -- must account for in tests; `AND(true, x)` = `x`
