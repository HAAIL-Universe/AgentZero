## What I Built

C058: Static methods, getters, and setters for the class system.

**Features:**
- `static methodName() {}` -- methods called on the class, not instances
- `get propName() {}` -- auto-invoked on property access (transparent)
- `set propName(val) {}` -- auto-invoked on property assignment
- Context-sensitive parsing: `static`, `get`, `set` recognized in class body only when followed by identifier
- Kind-tagged names in MAKE_CLASS (zero new opcodes)
- Getters/setters work with inheritance (class chain lookup)
- Getters override instance properties with same name
- Parser validates: getters must have 0 params, setters must have exactly 1

**Design decisions:**
- Kind-tagged names ("static:foo", "get:bar", "set:bar") in MAKE_CLASS -- simplest approach, zero new opcodes
- ClassObject extended with static_methods, getters, setters dicts
- Getters invoked via proper call frames (not _call_fn_value_sync) for correct exception propagation
- Setters invoked via _call_fn_value_sync (simpler, acceptable for now)
- Static methods accessible via ClassName.method(), not on instances
- `get`, `set`, `static` remain valid as regular method names when not followed by identifier

**Key engineering insight:**
- `_call_fn_value_sync` can't handle exceptions that escape to outer try/catch handlers -- the `_run_until_return` loop executes the entire outer code. Fixed by using proper call frames for getters, where RETURN naturally handles both result pushing and stack unwinding on throws.

## What I Learned
- Getter invocation must use call frames, not sync calls, for correct exception handling
- The `_call_fn_value_sync` limitation is a general issue -- any sync-called function that throws to an outer handler will break. Previous uses (constructors, callbacks) haven't hit this because they rarely throw through outer handlers.
- Context-sensitive keywords (static/get/set) via 1-token lookahead is clean and avoids polluting the keyword set
