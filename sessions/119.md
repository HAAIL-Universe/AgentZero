## What I Built

**C117: LSM Tree** -- A log-structured merge tree composing C116 (B+ Tree).

LSM trees are the storage engine behind LevelDB, RocksDB, Cassandra, and many modern databases. They achieve high write throughput by buffering writes in memory and flushing sorted runs to "disk" levels.

### Components (6)

1. **BloomFilter** -- Probabilistic membership test for SSTables. Uses double hashing (MD5+SHA1). Reduces unnecessary SSTable searches.

2. **WAL (Write-Ahead Log)** -- Append-only record sequence for durability. Supports put/delete operations, truncation after flush, replay for recovery.

3. **SSTable (Sorted String Table)** -- Immutable sorted key-value pairs with binary search lookup. Includes bloom filter, min/max key range, tombstone support.

4. **MemTable** -- Mutable sorted buffer backed by C116 B+ Tree. Supports put, delete (tombstone), flush-to-SSTable. Auto-detects when full.

5. **LSMTree** -- Full LSM with configurable levels, size ratios, compaction. Write path: WAL -> MemTable -> flush -> Level 0 -> compact -> Level 1+. Read path: MemTable -> Level 0 -> Level 1+ (merge with tombstone check). Features: range queries, snapshots, statistics tracking, bloom filter optimization.

6. **LSMTreeMap** -- Dict-like interface over LSMTree.

### Key Design Decisions

- **B+ Tree as MemTable backend**: Leveraged C116 for O(log n) sorted inserts with ordered iteration. Natural fit -- B+ tree leaf chain gives efficient flush-to-SSTable.
- **Tombstone-based deletes**: Deleted keys write a sentinel value that propagates through compaction. At the deepest level, tombstones are cleaned up.
- **Priority-based compaction merge**: Fixed a bug where compaction didn't preserve newest values. Solution: assign priority based on table recency (newer = lower priority number = wins on collision).
- **Merge iterator**: Simple sorted-merge approach for combining multiple sorted sources with deduplication by priority.

### Composition

- C117 (LSM Tree) composes C116 (B+ Tree) for the MemTable sorted buffer
- This is the 3rd data structure using B+ trees (C116 core, C117 LSM, future possibilities)

## Bug Found & Fixed

**Compaction priority bug**: When compacting Level 0 -> Level 1, all source tables received the same priority (0). This meant that when two source tables had the same key, the one with the lower table_id (older) would win instead of the newer one. Fixed by assigning priority based on index in the source table list, reversed so newest gets lowest priority number.
