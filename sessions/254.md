## What Was Built

### P1: Model-Initiated Tool Execution (agent_zero_server.py)

The model (Qwen3-14B via vLLM) can emit `<tool>name</tool>\n{json}` during generation. Previously these were stripped and discarded. Now:

- **`_extract_model_tool_calls()`** -- Parses tool blocks from raw model output
- **`_text_has_pending_tool_tag()`** -- Detects incomplete tool tags during streaming (buffers instead of sending partial tags to client)
- **`_execute_model_tool_calls()`** -- Runs extracted tools via `execute_tool()`, emits runtime events
- **`_build_tool_use_prompt_section()`** -- Injects tool-use instructions into the system prompt
- **Tool loop** -- Streaming generation wraps in a loop (max 3 rounds). If the model emits a tool call, execution happens, results are injected into context, and generation continues.

Key design: the tool results are appended as a system-injected message with structured data, then the model continues. The client only sees clean text -- tool blocks are stripped from the stream.

**20 tests** (test_model_tool_interception.py)

### P2: Context Compression (context_manager.py)

New module replacing the hardcoded 3000-char truncation with intelligent compression:

- **Token estimation** -- ~4 chars/token heuristic for fast estimation
- **Compression threshold** -- Triggers at 70% of 32K context window
- **Protected turns** -- Last 6 messages always preserved uncompressed
- **Extractive summarization** -- Deterministic topic/goal/question extraction from older messages
- **Summary injection** -- Compressed summary replaces old messages as a single context message
- **Compression artifacts** -- Summary logged to reasoning run for transparency

**30 tests** (test_context_manager.py)

### P3: Predictive Scenario Engine -- Verified End-to-End

No changes needed. The engine was already fully wired:
- `build_predictive_scenario_packet()` called during strategic reasoning
- Output feeds into simulate_outcomes, critique_paths, score_paths, decision_brief
- Scenario packet reaches cognitive blackboard and all agents
- Runtime events emit scenario metadata to frontend
- Behavioral/forecast/causal workers activate based on available data

**3 existing tests passing**

### P4: Selective Agent Orchestration (cognitive_runtime.py)

New `select_agents_for_turn()` function replaces the all-or-nothing worker activation:

- **casual_chat / simple_chat / self_introspection** -- No agents (handled by tools and direct generation)
- **Memory/recall turns** -- Fello only (empathy + context retrieval)
- **Memory + emotional turns** -- Fello + Othello (empathy + ethics)
- **Simple strategic** -- Fello + Othello (skip Prefrontal planning)
- **Complex strategic** -- Full pipeline: Fello + Othello + Prefrontal + Pineal

Complexity determined by: constraints count, time pressure, emotional register, priority count, word count, question presence.

`run_cognitive_runtime()` now accepts `agent_selection` parameter. Server passes selected agents from the routing function.

**16 tests** (test_selective_agents.py)

### P5: A2 Verification Interface (tool_registry.py, tool_runtime.py)

Two new tools for Agent Zero to interact with A2:

- **`analysis.request`** -- Sends a mission to A2 via the message queue. Includes task description, tool hint, and priority. A2 picks it up in its next session and runs the requested analysis.
- **`analysis.results`** -- Checks Agent Zero's MQ inbox for completed findings from A2.

Both tools use the existing `tools/mq.py` infrastructure via subprocess. Results are asynchronous by design -- A2 runs independently.
