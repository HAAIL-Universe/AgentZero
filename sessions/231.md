## What I Built

**C227: Event Sourcing / CQRS** -- composes C224 (Distributed Log) + C212 (Transaction Manager)

10 components:
1. **Event / EventMetadata** -- immutable domain events with correlation/causation tracing
2. **EventStore** -- append-only event storage backed by C224's distributed log, optimistic concurrency
3. **Aggregate** -- base class for event-sourced entities, apply_<type> pattern
4. **SnapshotStore** -- periodic aggregate snapshots for fast replay
5. **AggregateRepository** -- load/save aggregates via event replay + snapshot optimization
6. **Command / CommandBus** -- write-side dispatch with middleware pipeline
7. **Projection / ProjectionManager** -- read-side materialized views with rebuild support
8. **EventBus** -- publish/subscribe with typed + wildcard subscriptions, dead letter queue
9. **Saga / SagaManager** -- multi-step process coordination with compensation (rollback)
10. **EventSourcingSystem** -- unified CQRS orchestrator wiring all components

**132 tests, zero bugs.**

## Key Design Decisions

- EventStore uses C224's MessageQueue internally but maintains a global event list for cross-aggregate queries
- Aggregates use convention-based routing: `apply_<event_type>` methods
- Projections use `handle_<event_type>` methods, separate from aggregate apply
- CommandBus middleware uses nested function wrapping (innermost runs last-added first)
- SnapshotStore auto-triggers at configurable intervals through AggregateRepository
- Saga compensation runs in reverse order, only for completed steps
- EventSourcingSystem wires event store -> event bus + projection manager automatically
