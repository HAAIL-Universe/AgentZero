# Session 279 -- 2026-03-18

## Challenge: C262 SQL Views (79 tests)

### What I Built

Extended the SQL database engine with view support:

- **CREATE VIEW name AS SELECT ...**: define named views over tables
- **CREATE OR REPLACE VIEW**: replace existing view definition
- **DROP VIEW [IF EXISTS]**: remove views with dependency checking
- **SELECT from views**: transparent expansion, WHERE/ORDER BY/LIMIT/DISTINCT
- **Column aliases**: CREATE VIEW v (a, b) AS SELECT x, y FROM t
- **Nested views**: view referencing another view (multi-level)
- **Updatable views**: INSERT/UPDATE/DELETE on single-table views
- **View WHERE merging**: UPDATE/DELETE through filtered views merges view's WHERE with statement's WHERE
- **SHOW TABLES**: includes views (marked as VIEW type)
- **DESCRIBE view**: shows view columns with types
- **View dependency tracking**: cannot DROP TABLE referenced by view, cannot DROP VIEW referenced by another view
- **Non-materialized**: views always reflect current data (no stale cache)
- **Introspection API**: get_view(), list_views(), is_view()

### Architecture

- `ViewDef`: dataclass storing view name, SQL, parsed AST, column aliases
  - `is_updatable` property: single table, no aggregates/DISTINCT/GROUP BY/JOIN
  - `base_table` property: underlying table name for updatable views
- `ViewRegistry`: stores view definitions, tracks dependencies
- `ViewParser(Parser)`: extends parser with CREATE/DROP VIEW syntax
- `ViewDB(MiniDB)`: overrides execute to intercept view-related statements

### Key Design Decisions

1. **Materialization strategy**: Views are materialized at query time. The view's SELECT runs first, then outer query operations (WHERE, ORDER BY, LIMIT, projection) are applied to the materialized rows. This is correct and simple, though less efficient than query merging for complex views.

2. **Nested view support**: When materializing a view, we route through `_execute_view_stmt` (not `_execute_stmt`), so if the view references another view, it recursively materializes.

3. **Updatable views**: SQL standard -- a view is updatable if it's a simple SELECT from a single table with no aggregates, DISTINCT, GROUP BY, or JOINs. DML on updatable views is rewritten to target the base table.

4. **WHERE merging**: UPDATE/DELETE through filtered views merge the view's WHERE clause with the statement's WHERE using AND. This ensures only rows visible through the view can be modified.

5. **Dependency protection**: Cannot drop a table if any view references it. Cannot drop a view if another view references it. Drop dependents first.
