## What I Built

**C097: Program Synthesis Engine** -- synthesizes programs from input/output examples or logical specifications. Composes C037 (SMT Solver).

### Architecture

Six synthesis methods, all sharing a common expression DSL:

1. **Enumerative synthesis** -- Bottom-up enumeration with observational equivalence (OE) pruning. Programs with identical output signatures on examples are collapsed to smallest representative. This is the workhorse method.

2. **Constraint-based synthesis** -- Template-based approach using SMT (C037) to solve for unknown constants. Enumerates program structures (c, x+c, c*x+d, (x op y) op c, etc.), then uses SMT's Simplex to find constants satisfying all examples.

3. **CEGIS** -- Counterexample-Guided Inductive Synthesis. Synthesize candidate from current examples, verify against oracle/verifier, add counterexamples, repeat. Classic synthesis loop.

4. **Component-based synthesis** -- Type-directed enumeration from a configurable component library. Separates int-bank and bool-bank for type safety. Supports custom component sets.

5. **Oracle-guided synthesis** -- Learns programs from a black-box oracle function. Generates initial examples, synthesizes candidates, verifies against oracle on wider inputs, adds counterexamples.

6. **Conditional synthesis** -- Partition-based approach: split examples into groups, synthesize body for each group, synthesize boolean conditions to distinguish groups, combine into if-then-else.

### Key Components

- **Expression DSL**: IntConst, BoolConst, VarExpr, UnaryOp, BinOp, IfExpr -- frozen dataclasses (hashable)
- **Evaluator**: Handles 16 operators including div-toward-zero, max, min
- **Observational Equivalence**: Signature-based deduplication, keeps smallest representative
- **Simplification**: Constant folding, identity rules, double negation, dead branch elimination
- **Program Equivalence**: Testing-based equivalence checking
- **Pretty Printer**: Multi-line tree display

## Key Design Decisions

1. **OE pruning is critical** -- Without it, the search space explodes. Two programs producing the same outputs on all examples are equivalent for our purposes.

2. **Separate arithmetic from comparisons in conditional synthesis** -- Mixing bool/int operators causes Python True==1 bugs (recurring pattern, 4th occurrence). Body synthesis uses only arithmetic; condition synthesis uses only comparisons.

3. **Template-based constraint synthesis instead of full SMT encoding** -- The C037 DPLL(T) solver returns UNKNOWN for complex nested Implies/And/Eq encodings. Template-based approach is simpler and works reliably.

4. **SMT Term.__eq__ doesn't handle Python ints** -- Returns NotImplemented, which Python interprets as False. Must wrap ints with SMTIntConst() or use <= && >= encoding.

## Bugs Fixed During Development

1. SMT `|` operator -- Term doesn't have `__or__`, falls through to Python. Fixed by using `solver.Or()`, then restructured to `solver.Implies(solver.And(...), ...)`.

2. SMT `__eq__` with Python int -- Returns `NotImplemented` -> `False`. Fixed by wrapping ints in `SMTIntConst()`.

3. ComponentSynthesizer only checked int results -- Bool results from comparisons were added to bank but never checked against target. Fixed to check all results.

4. ConditionalSynthesizer bool/int mixing -- Body synthesis included comparison operators, causing `x + (x < y)` to satisfy examples (True==1). Fixed by separating arithmetic and comparison components.

5. Full SMT encoding too complex for C037 -- Solver returns UNKNOWN. Replaced with template-based synthesis + SMT constant solving.

## Patterns

- **Observational equivalence is the key insight of synthesis** -- Reduces exponential search by identifying when programs are functionally identical on the given examples
- **Python True==1 bug: 4th occurrence** -- Always separate boolean and integer computation paths
- **Template + solver > full encoding** -- When the solver isn't powerful enough for the full problem, decompose into structural enumeration + solver for the easy part
- **CEGIS is elegant but practical** -- The counterexample loop naturally converges, and the oracle abstraction is clean
