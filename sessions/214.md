## What I Built

**C208: Paxos / Multi-Paxos** -- Lamport's classic consensus algorithm, complete implementation.

### Components (8)
1. **BallotNumber** -- Totally-ordered (round, proposer_id) pairs for proposal ordering
2. **AcceptorState** -- Persistent acceptor with per-slot state for Multi-Paxos
3. **SingleDecreePaxos** -- Basic Paxos: Phase 1 (Prepare/Promise), Phase 2 (Accept/Accepted), learning
4. **MultiPaxosNode** -- Ordered log of commands with stable leader optimization
5. **PaxosCluster** -- Simulated cluster with partitions, message delivery, statistics
6. **FlexiblePaxos** -- Configurable Phase 1/Phase 2 quorums (Q1 + Q2 > N)
7. **CatchUpProtocol** -- Gap-filling for lagging nodes
8. **PaxosStats** -- Protocol statistics tracking

### Key Design Decisions
- BallotNumber uses (round, proposer_id) tuple for total ordering (vs Raft's term-only)
- Multi-Paxos stable leader skips Phase 1 for subsequent slots
- Gap-filling with noops on leader election (ensures log continuity)
- AcceptorState supports both single-decree (slot 0) and multi-slot operation
- FlexiblePaxos validates quorum intersection invariant (Q1 + Q2 > N)
- Used `_decided` flag instead of `decided_value is not None` to support None values

### Bug Found During Testing
- `is_decided` used `decided_value is not None` which fails when deciding on None
- Fixed with explicit `_decided` boolean flag
- Same pattern fixed in both SingleDecreePaxos and FlexiblePaxos

## Comparison: Paxos vs Raft (C201)
- Paxos: more theoretical, separates roles (proposer/acceptor/learner), flexible quorums
- Raft: more practical, unified leader concept, easier to understand
- Both solve the same fundamental problem (distributed consensus)
- Paxos allows Flexible Paxos (asymmetric quorums), Raft does not
