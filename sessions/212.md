### Architecture

The distributed KV store partitions the keyspace using consistent hashing, replicates each partition via a Raft consensus group, tracks versions with vector clocks for conflict detection, and uses gossip for cluster membership. It supports:

- **4 consistency levels**: ONE, QUORUM, ALL, STRONG (linearizable via Raft)
- **3 consistency modes**: STRONG, EVENTUAL, SESSION (read-your-writes)
- **Operations**: GET, PUT, DELETE, CAS, SCAN, BATCH
- **Features**: read repair, hinted handoff, anti-entropy sync, tombstone GC
- **REST API**: full CRUD + cluster management endpoints

### Components (6)
1. **Partition** -- key range data layer with version vectors, siblings, tombstones
2. **PartitionStateMachine** -- Raft state machine wrapping Partition
3. **ReplicaGroup** -- Raft cluster managing one partition
4. **ClusterNode** -- physical node hosting multiple partition replicas
5. **KVCluster** -- the full distributed system orchestrating all components
6. **KVHttpHandler** -- REST API adapter

## Key Decisions

1. **Raft for strong consistency, VV for eventual** -- dual mode depending on consistency level
2. **committed_results pattern** -- Raft stores results by request_id, not via outbox messages
3. **Single-node handling** -- explicitly call _advance_commit_index when no peers
4. **None command handling** -- Raft no-op entries (leader election) return ok

## Bugs Found & Fixed (During Development)

1. `_outbox` vs `outbox` -- Raft uses public `outbox` attribute
2. `None` command in state machine -- Raft election creates no-op log entries
3. `_handle_node_info` reference -- method was named `_handle_node_info_id`
4. Commit timeout -- needed to use `committed_results` not outbox scanning
5. Single-node Raft -- no peers means no `_advance_commit_index` trigger

All fixed before final test run. Zero bugs in final run.
