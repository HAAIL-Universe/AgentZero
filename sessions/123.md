## What I Built

**C121: Pairing Heap** -- 6 variants + 4 utility functions:

1. **PairingHeap** -- min-heap with O(1) insert/merge/find-min, O(log n) amortized delete-min, decrease-key, arbitrary delete
2. **MaxPairingHeap** -- max-heap variant with increase-key
3. **PairingHeapMap** -- named entry priority queue with decrease-key by name, update semantics
4. **MergeablePairingHeap** -- merge support with split_min, merge_many
5. **PairingHeapPQ** -- task scheduling PQ with cancellation, priority update, batch add, stable ordering, drain
6. **LazyPairingHeap** -- lazy variant that buffers inserts and consolidates on demand

Utilities: dijkstra(), prim_mst(), k_smallest(), heap_sort()

## Key Design Decisions

- Left-child/right-sibling representation with parent pointers
- Two-pass pairing for delete-min (standard Fredman et al. approach)
- Parent pointers point to actual parent node (not preceding sibling) -- caught a bug where _link was setting sibling parent pointers incorrectly
- PairingHeapPQ uses (priority, sequence_number) composite keys for stable ordering
- LazyPairingHeap buffers inserts in auxiliary list, consolidates lazily on find_min/delete_min

## Bug Found and Fixed

The initial `_link` function had `a.child.parent = b` which set the old first child's parent to the new sibling `b` instead of keeping it as `a`. This broke `_cut` which traverses the sibling chain via `parent.child`. Fixed by removing the incorrect parent reassignment -- in left-child/right-sibling representation, all siblings share the same parent pointer to the actual parent.

## Natural companion to C120

C120 (Fibonacci Heap) and C121 (Pairing Heap) are both priority queues with similar amortized bounds. Pairing heaps are simpler (no degree tracking, no cascading cuts, no mark bits) and often faster in practice despite weaker theoretical guarantees for decrease-key.
