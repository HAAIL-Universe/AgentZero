## Design

Async generators are functions that can both `yield` values and `await` promises. Syntax: `async fn* name() { ... }`.

### Key decisions:
- `AsyncGeneratorObject` dataclass -- similar to GeneratorObject but with `next_promise` field
- `next(agen)` returns a PromiseObject that resolves to the yielded value
- `await next(agen)` unwraps the promise to get the value
- YIELD inside async gen: resolves the pending next() promise with the yielded value
- AWAIT inside async gen: suspends execution, registers callbacks on the inner promise
- RETURN/end: resolves next() promise with None, marks generator done
- THROW without handler: rejects next() promise

### Implementation changes:
1. **Parser**: `_is_async_fn_decl` recognizes `async fn* name`. `async_fn_decl` consumes optional star.
2. **AST**: `AsyncFnDecl` gets `is_generator: bool` field to track `fn*` syntax explicitly
3. **Compiler**: `compile_AsyncFnDecl` checks both `node.is_generator` and `_ast_contains_yield` to set FnObject flags
4. **VM**: New `AsyncGeneratorObject`, `_create_async_generator`, `_next_async_generator`, `_resume_async_generator`, `_resume_async_generator_with_error`, `_resume_async_generator_continue` methods
5. **CALL dispatch**: All 3 dispatch points (sync call, CALL opcode, CALL_SPREAD) check `is_generator and is_async` before plain `is_generator`
6. **Async queue**: Now handles 4-tuple entries `('agen', agen, value, error)` alongside regular 3-tuples
7. **Builtins**: `next()` handles AsyncGeneratorObject, `type()` returns "async_generator", `_format_value` formats it
8. **ITER_PREPARE**: Handles AsyncGeneratorObject (drains async queue between yields)

### One bug found during testing (test-time, not implementation):
- Empty async generators (`async fn* gen() {}`) were treated as regular async functions because `_ast_contains_yield` returned False. Fix: store `is_generator` flag explicitly in AST from parser's `fn*` syntax.
