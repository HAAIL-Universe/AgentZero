## What I Built

C212: Transaction Manager -- ACID transaction manager with MVCC, WAL, savepoints, and multiple isolation levels.

### Components
- **WAL (Write-Ahead Log)**: Durable log with LSN tracking, crash recovery, checkpoint support
- **MVCC (Multi-Version Concurrency Control)**: Row versioning with snapshot isolation, visible version selection
- **Snapshot**: Point-in-time view with active-tx tracking for correct visibility
- **LockManager**: Row-level and table-level locking with deadlock detection, lock upgrades
- **Transaction**: Full lifecycle with insert/update/delete/read, savepoints, commit/rollback
- **TransactionManager**: Orchestrates transactions, snapshots, WAL, locking, recovery, vacuum
- **TransactionalDatabase**: High-level convenience API with auto-retry on serialization failure

### Isolation Levels
- READ_UNCOMMITTED: Sees all uncommitted changes (dirty reads)
- READ_COMMITTED: Fresh snapshot per read (no dirty reads)
- REPEATABLE_READ: Snapshot at begin time (no phantoms)
- SERIALIZABLE: Snapshot + write-write + read-write conflict detection

### Key Design Decisions
1. **MVCC over pure locking**: Readers never block writers. Snapshot isolation handles read consistency.
2. **Snapshot at begin time** for REPEATABLE_READ/SERIALIZABLE (not lazy on first read)
3. **No read locks**: MVCC handles read isolation; only writes acquire EXCLUSIVE locks
4. **Write-write conflict detection** checks both active txs AND recently committed txs (prevents lost updates)
5. **Savepoints** via write_set snapshots + undo operations (not full table cloning)

### Bugs Found and Fixed (during development, before final run)
1. Lazy snapshot creation caused REPEATABLE_READ to see committed changes -- fixed by taking snapshot at BEGIN
2. Shared read locks caused lock conflicts in concurrent SERIALIZABLE scenarios -- removed read locks, rely on MVCC
3. Write-write conflict detection only checked active txs, missed committed ones -- added committed_since check
