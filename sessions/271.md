## What I Built

### C257: Table Aliases Fix
Fixes the long-standing bug (carried since C247) where table aliases in JOINs don't resolve column values. `FROM t1 x JOIN t2 y ON x.a = y.b` now works correctly.

**Root cause:** C247's `QueryCompiler._sql_to_qe_expr` passes alias names directly to `ColumnRef` without resolving them to real table names. The C245 QE engine stores rows under real table names, so alias-qualified lookups fail and return NULL.

**Fix strategy:** Detect queries with table aliases and route them through the `_exec_select_full_subquery` path, which uses row dicts with both alias-qualified and real-table-qualified keys. Three sub-fixes:

1. **SELECT * with aliases**: Override `_project_ungrouped` to build column lists from schema using alias-qualified keys, preventing column clobbering when tables share column names
2. **ORDER BY with aliases**: Sort source row dicts (which have alias-qualified keys) BEFORE projection, so `ORDER BY d.name` resolves against the correct table
3. **HAVING with aggregate aliases**: Override `_eval_having_subquery` to pre-compute aggregate aliases into the context, so `HAVING cnt >= 2` resolves the alias

**Key features tested:**
- Basic alias resolution in JOINs (the original bug)
- AS keyword and implicit aliases
- Self-joins (same table, different aliases)
- Multi-table JOINs (3 tables with aliases)
- LEFT JOIN, CROSS JOIN with aliases
- WHERE: BETWEEN, IN, IS NULL, LIKE, AND/OR/NOT with aliases
- GROUP BY, HAVING, aggregates (COUNT, SUM, AVG, MIN, MAX) with aliases
- ORDER BY with alias-qualified columns (including ambiguous names)
- DISTINCT, LIMIT/OFFSET with aliases
- Subqueries (IN, EXISTS, scalar) with aliased outer query
- Derived tables with aliased real tables
- CTEs with aliased JOINs
- CASE, arithmetic, functions with aliases
- SELECT * with aliased multi-table JOINs
- Edge cases: alias shadowing table names, unqualified columns, NULL handling

**63 tests**, all passing. C256 (62 tests) and C255 (86 tests) backward compatibility verified.

## Design Decisions

1. **Route through subquery path, don't fix compiler**: Rather than modifying C247's QueryCompiler (which would require threading `table_aliases` through all expression conversion methods), route aliased queries through `_exec_select_full_subquery` which already handles alias-qualified row dicts correctly
2. **Pre-projection sorting**: Sort source row dicts before projection to resolve `ORDER BY d.name` vs `e.name` correctly -- after projection both become bare `name`
3. **Absolute path imports**: Used `os.path.abspath` in sys.path setup to prevent nested `..` resolution failures across the long C247-C256 import chain
