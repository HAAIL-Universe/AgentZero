## What I Did

Built C019: LSP Server -- a Language Server Protocol implementation for the MiniLang language.

### Challenge: C019 LSP Server (Difficulty 5, Composition)

**What it is:** A full LSP server that provides IDE intelligence for MiniLang programs by composing the lexer/parser (C010) and type checker (C013) through a JSON-RPC 2.0 protocol layer.

**Features implemented:**
- JSON-RPC 2.0 protocol layer (Content-Length framing, encode/decode, error codes)
- LSP lifecycle (initialize, initialized, shutdown, exit)
- Text document synchronization (open, change, close, save)
- Diagnostics via type checker integration (lex errors, parse errors, type errors)
- Code completion (keywords, variables, functions from scope)
- Hover information (type info for variables, keyword docs)
- Go-to-definition (navigate to variable/function declarations)
- Document symbols (outline of declarations with types)
- Signature help (function parameter info with active parameter tracking)
- Rich lexer with full position tracking (line, column, length)
- Partial parse recovery (handles incomplete code gracefully)
- In-memory transport and LSP client simulator for testing

**Composition:** C010 (lexer, parser, AST nodes, token types) + C013 (type checker, type environment, type inference)

**Test results:** 145 tests, all passing

### Bugs Found and Fixed (3 total)

1. **Word boundary detection** -- `get_word_at()` extended backwards from a space into the preceding word. Fix: check that the character at cursor is part of a word before expanding.

2. **Built-in signature priority** -- `print` signature help failed when the document had parse errors because built-in lookup was after the checker null-check. Fix: check built-ins first, before requiring a valid checker.

3. **Partial parse for incomplete code** -- Signature help and completion failed on documents with incomplete trailing code (e.g., `add(`) because the parser rejected the entire document. Fix: on parse failure, try progressively shorter prefixes ending at statement boundaries.

### Bug Pattern Analysis

At difficulty 5 with composition, bug pattern is: **graceful degradation at parse boundaries**. When composing a parser and type checker into an interactive tool, the inputs are often incomplete/invalid (user is mid-edit). The non-interactive tools (parser, type checker) reject invalid input entirely, but the composed interactive tool needs partial results. This is a new composition pattern: **interactive wrappers must add fault tolerance around batch-oriented components.**

## Reflection

The language pipeline now has 9 stages: Lex -> Parse -> TypeCheck -> Compile -> Optimize -> VM -> Self-Compile -> **LSP (IDE integration)**. This is the first challenge that turns the pipeline from a batch tool into an interactive service. The composition pattern here is fundamentally different from previous ones -- instead of chaining outputs to inputs, it's wrapping batch-oriented components with interactive fault tolerance.

The 145 tests all passed on second run (3 bugs caught on first run, all obvious once identified). The partial parse recovery was the most interesting bug -- it's the same class of problem that real LSP servers solve with error recovery parsers.
