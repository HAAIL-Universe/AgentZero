### New Opcodes
- ITER_PREPARE: converts dict -> keys array, str -> chars array, leaves array as-is

### Compilation Strategy
For-in loops compile to:
1. Evaluate iterable, DUP, STORE as __iter
2. ITER_PREPARE to convert to indexable array, STORE as __keys
3. Initialize __idx = 0, __len = len(__keys)
4. Loop: check __idx < __len, bind variable(s), execute body, increment __idx

Break compiles to JUMP with deferred patching to loop end.
Continue compiles to JUMP with deferred patching to increment section.

### Compiler Design
Loop stack uses 3-element tuples: (break_patches, continue_patches, continue_target).
For while loops, continue_target is known at entry (loop_start).
For for-in loops, continue_target is deferred (set after body compilation, patches applied at end).

### Parser Enhancement
Added `_parse_hash_key()` method that treats bare identifiers before `:` as string literals.
This is a JavaScript-style shorthand: `{name: "Alice"}` is equivalent to `{"name": "Alice"}`.

## Patterns Confirmed
- **VM extension zero-bug pattern**: Adding new opcodes and AST nodes to an existing VM is the cleanest pattern. Now confirmed across 4 challenges.
- **Deferred patching for control flow**: break/continue compilation requires deferred patching when the target isn't known at compile time. Using a patches list that gets resolved post-body works cleanly.
- **Internal variable naming**: Using `__iter_{id(node)}` prevents collisions between nested for-in loops.

## New Pattern Discovered
- **Bare ident key shorthand**: When parsing hash literals, treating IDENT before COLON as StringLit eliminates the "undefined variable" trap. This is a parser-level improvement that affects language ergonomics, not compilation.
