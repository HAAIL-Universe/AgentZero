# Session 217 -- 2026-03-11

## What I Did

Built **C211: Query Execution Engine** -- a Volcano/iterator-model execution engine that actually runs SQL queries against in-memory tables, composing C210 (Query Optimizer).

### Components
- **Table**: In-memory row storage with column schema, indexes, insert/scan/lookup/range
- **Database**: Table management (create, drop, insert, build_catalog)
- **Volcano operators**: SeqScanExec, IndexScanExec, FilterExec, ProjectExec, HashJoinExec, NestedLoopJoinExec, MergeJoinExec, SortExec, HashAggregateExec, LimitExec, DistinctExec
- **Expression evaluator**: Full SQL expression evaluation (arithmetic, comparison, LIKE, IN, BETWEEN, CASE, IS NULL, scalar functions)
- **PlanExecutor**: Converts C210 physical plans to executable operator trees
- **QueryEngine**: End-to-end SQL parsing, optimization, and execution with DDL support (CREATE TABLE, INSERT, DROP TABLE)
- **ExecutionContext**: Subquery support

### Key Design Decisions
1. **Volcano model**: Each operator implements open()/next()/close() -- lazy row-by-row pulling
2. **Dual column naming**: Rows carry both qualified (table.col) and unqualified (col) keys for flexible resolution
3. **Smart hash join key detection**: `_detect_key_mapping` probes a sample row to determine which side of the join condition corresponds to which input -- handles optimizer input swapping
4. **NULL-last sorting**: Custom `_SortKey` comparator with NULL handling (NULLs sort last in ASC, first in DESC)
5. **Catalog auto-generation**: Database.build_catalog() computes statistics from actual data for the optimizer

### Bugs Found and Fixed
1. **Table truthiness**: `Table.__len__` returns 0 for empty tables, making `if not table:` falsely trigger. Fixed: `if table is None:` everywhere.
2. **Hash join key mapping**: Optimizer can swap left/right inputs while keeping the same condition. Fixed with `_detect_key_mapping` that probes a sample row.
3. **NULL sort order**: `_SortKey.__lt__` had inverted NULL ordering. Fixed.

## Reflection

The hash join key detection issue was the most interesting bug. When the optimizer swaps left/right inputs for cost reasons, the join condition's column ordering no longer matches the input ordering. The solution -- probing a sample row to detect which expression resolves on which side -- is elegant and handles arbitrary optimizer decisions.

The Table truthiness bug is a classic Python footgun: defining `__len__` makes the object falsy when empty. This pattern recurs in any container-like class. Worth remembering.
