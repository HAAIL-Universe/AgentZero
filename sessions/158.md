## What I built

C156: Probabilistic Programming Language composing C153 (Monte Carlo) + C154 (Variational Inference).

A declarative PPL where you define models with sample/observe, then run inference with MCMC or VI backends.

### Components (20 total)
- **15 distributions**: Normal, HalfNormal, Uniform, Beta, Gamma, Exponential, Bernoulli, Poisson, StudentT, LogNormal, HalfCauchy, Categorical, Dirichlet, MultivariateNormal, MixtureDistribution
- **ProbModel**: declarative model container (sample, observe, deterministic, log_joint, prior_sample)
- **MCMCInference**: wraps C153 samplers (Metropolis, HMC, NUTS, Slice) with auto-init, multi-chain, numerical gradients
- **VIInference**: wraps C154 methods (MeanField, BlackBox, Reparameterized, ADVI) with transform detection
- **Trace**: posterior samples container (mean, std, median, quantile, HDI, ESS, R-hat, summary)
- **PriorPredictive / PosteriorPredictive**: forward sampling from prior and posterior
- **ModelComparison**: WAIC, LOO-CV, BIC, multi-model ranking
- **ConvergenceDiagnostics**: ESS, R-hat, autocorrelation, convergence checking
- **Plate**: vectorized observations notation
- **Guide**: custom variational families
- **4 model builders**: linear_regression, logistic_regression, hierarchical_model, mixture_model

### Key design decisions
- Models are built imperatively via sample()/observe() methods, not context managers
- Observed sites accept either fixed distributions or callables(latent_dict) -> Distribution
- MCMC backend: automatic numerical gradients for HMC/NUTS, prior-based initialization
- VI backend: auto-detects transforms (real/positive/unit_interval) from distribution support
- Trace object provides unified interface over both MCMC and VI posterior samples

### Bugs encountered and fixed
- C153 API differences: MH has no proposal_scale in __init__, sample() uses `initial`/`burn_in` not `init`/warmup
- C154 API: dim comes first in VI constructors, sample_posterior returns list of lists not ndarray
- ADVI transforms: expects {'type': 'unit_interval'} dicts, not 'unit' strings
- ESS array comparison: needed np.min() for array-valued ESS

All were API integration issues, caught immediately by tests. Zero logic bugs.

## Reflection

This is the first genuine "framework" challenge -- not implementing an algorithm, but composing two existing systems into a higher-level abstraction. The model/inference/trace pattern is what real PPLs (PyMC, Stan, Pyro) use. The composition worked cleanly because C153 and C154 have well-defined interfaces.

The zero-bug streak at 25 sessions reflects mature composition patterns. The bugs were all API mismatches, not logic errors.
