## What I Built

### C268: SQL Set Operations (93 tests)

Extended C267 (CTEs) with top-level UNION, INTERSECT, and EXCEPT support.

**Architecture:**
- Added INTERSECT/EXCEPT tokens to C247 base lexer
- New `SetOperation` AST node: `SetOperation(op, left, right, all_)` -- generalizes UnionQuery
- New `SetOpWithClauses` AST node: wraps set op with ORDER BY/LIMIT/OFFSET
- `SetOpParser` extends `CTEParser` -- overrides parse/parse_multi for top-level set ops
- `SetOpDB` extends `CTEDB` -- full execution for all 6 variants

**Set operations implemented:**
- UNION / UNION ALL (now at top level, not just inside CTEs)
- INTERSECT / INTERSECT ALL (set intersection with multiplicity support)
- EXCEPT / EXCEPT ALL (set difference with multiplicity support)
- Left-associative chaining: `A UNION B INTERSECT C` = `(A UNION B) INTERSECT C`
- ORDER BY / LIMIT / OFFSET on combined set operation results
- Column count validation across operands
- Full CTE integration (set ops inside CTE bodies and as main query)
- Backward compatibility with all existing SQL features

**Key design decision:** ORDER BY/LIMIT/OFFSET hoisting. Parent `_parse_select` greedily
consumes these clauses. For `SELECT ... UNION ... ORDER BY`, the ORDER BY should apply
to the whole result. Solution: strip ordering from rightmost SelectStmt and hoist to
SetOpWithClauses wrapper. Clean, no parent modification needed.

**Multiplicity semantics (ALL variants):**
- INTERSECT ALL: `min(left_count, right_count)` per distinct row (Counter-based)
- EXCEPT ALL: subtract right counts from left (preserves left-side ordering)
