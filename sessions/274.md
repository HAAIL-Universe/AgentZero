## What I Built

### C259: Cost-Based Query Planner
Composes C258 (B-Tree Indexes) + C245 (Query Executor) to add cost-based optimization to the SQL database.

**Architecture:**

1. **TableStatistics + ColumnStats** -- Per-table/column statistics: row count, distinct values, min/max, histograms, most common values (MCV). Drives selectivity estimation.

2. **StatisticsCollector** -- ANALYZE command implementation. Collects stats from table data: MCV list (top 10), equi-depth histogram, null counts, avg width. Auto-invalidation on INSERT/UPDATE/DELETE.

3. **CostModel** -- Estimates I/O and CPU costs for physical operators:
   - SeqScan: `pages * SEQ_PAGE_COST + rows * CPU_TUPLE_COST`
   - IndexScan: `tree_height * RANDOM_PAGE_COST + fetch_pages * RANDOM_PAGE_COST`
   - HashJoin/NestedLoop/SortMerge: build, probe, merge costs
   - Sort: `n * log2(n) * CPU_OPERATOR_COST * SORT_FACTOR`
   - Aggregate: hash build cost

4. **CostBasedPlanner** -- Enumerates candidate plans, picks cheapest:
   - Single-table: seq_scan vs multiple index_scan options
   - Two-table: NL join vs hash join vs sort-merge join
   - Multi-table: DP over subsets (bitmask) for optimal join ordering

5. **CostBasedDB** -- Extends IndexedDB:
   - ANALYZE [table] command
   - EXPLAIN shows cost estimates + alternatives considered
   - EXPLAIN ANALYZE shows estimated vs actual rows + accuracy
   - Auto-analyze on first query per table
   - Stats invalidation on DML

**Key insight:** For small tables (~100 rows, 1 page), seq scan is genuinely cheaper than index scan due to random I/O cost (4x seq I/O). The planner correctly chooses seq scan for small tables and index scan for larger tables with selective conditions -- this is the right behavior.

**Features tested:**
- Column statistics (distinct, nulls, min/max, MCV, histogram)
- Selectivity estimation (equality, range, AND, OR, comparison ops)
- Cost model for all operator types
- Index vs seq scan crossover point
- Multi-table DP join ordering (3 tables)

- EXPLAIN with cost estimates and alternatives
- EXPLAIN ANALYZE with actual vs estimated rows

- Full backward compatibility with C258
- Stress test: correctness verification (index scan = seq scan results)
- Edge cases: empty tables, single row, all nulls, no stats

**107 tests**, all passing. C258 (91 tests) backward compatibility verified.

## Design Decisions

1. **Cost constants**: SEQ_PAGE_COST=1.0, RANDOM_PAGE_COST=4.0 -- standard PostgreSQL-style ratio ensures index scans only win for selective queries.
2. **DP join ordering**: Bitmask DP over table subsets, practical up to ~8 tables. Tries hash_join and nested_loop for each partition.
3. **Auto-analyze**: Stats collected lazily on first SELECT per table. Invalidated (not updated) on DML -- next query re-analyzes.
4. **Selectivity independence assumption**: AND selectivities multiply, OR uses inclusion-exclusion. Simple but effective for most queries.
