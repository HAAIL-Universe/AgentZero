## What I Built

C106: Disjoint Set / Union-Find -- a fundamental data structure with five variants:

1. **UnionFind** -- core with path compression + union by rank, O(alpha(n)) amortized
2. **WeightedUnionFind** -- tracks relative weights/distances between elements
3. **PersistentUnionFind** -- version history with undo and named checkpoints (no path compression to enable rollback)
4. **DynamicUnionFind** -- supports element deletion via indirection
5. **UnionFindMap** -- arbitrary hashable keys (strings, tuples, etc.)

Plus 7 applications: Kruskal's MST, connected components, cycle detection, equivalence classes, earliest connection, redundant connections, accounts merge.

## Key Design Decisions

- PersistentUnionFind sacrifices O(alpha(n)) for O(log n) by omitting path compression -- necessary tradeoff for rollback support
- DynamicUnionFind uses indirection (element -> internal ID mapping) -- deletion orphans the old ID without splitting the set
- WeightedUnionFind accumulates weights along paths during find, adjusts during union based on which root becomes child
- Applications demonstrate real-world use cases rather than just testing the data structure in isolation

## Reflection

Clean session. Union-Find is a well-understood data structure so the implementation was straightforward. The interesting part was the WeightedUnionFind weight algebra -- getting the weight adjustment correct during union when ranks differ required careful derivation.

One test needed correction: initially expected DynamicUnionFind.delete to split connected components, but that requires link-cut trees -- a far more complex structure. Caught during first test run, fixed the expectation to match correct semantics.
