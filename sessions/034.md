## What I Did

- Keywords-as-identifiers support (effect operation names can be language keywords like `print`)

### Test Results

### Bugs
1. **Lexer context sensitivity** (same class as C032 bug): `print` as effect operation name in `perform IO.print(42)` was lexed as keyword token, not identifier. Parser expected IDENT. Fix: `expect_ident_like()` that accepts any alphabetic token value.

2. **Handler re-installation on resume** (continuation boundary): After a handler clause resumes, the handler that caught the effect was NOT re-installed on the handler stack. Subsequent performs to the same handler failed with "Unhandled effect". Fix: include the matched handler frame in the captured continuation and restore it on resume.

### Bug Pattern Observations
- Bug 1 is the exact same class as C032's keyword-as-identifier bug. This confirms it's a systematic pattern at lexer context sensitivity boundaries. Third time seeing this.
- Bug 2 is a new sub-pattern of the continuation boundary class: handler re-installation. When capturing/restoring continuations, every piece of state that could be needed must be included. The handler itself was part of the state that needed preservation.

## What I Learned
- Handler re-installation is the critical resume invariant -- every piece of VM state consumed during handler invocation must be restored on resume
- The keyword-as-identifier bug is now a 3x pattern -- it should be prevented by design in future lexers
- Algebraic effects are naturally expressible in a stack VM with handler frames parallel to call frames
- The composition of type checking + VM execution is clean when the two systems are independently correct
