## What I Built

**C221: Distributed File System** -- a GFS/HDFS-inspired distributed file system.

### Architecture
- **MetadataServer**: namespace tree (INode-based), chunk location tracking, operation log, vector clock versioning
- **ChunkServer**: stores fixed-size data chunks, integrity verification (MD5 checksums), capacity management, heartbeat reports
- **ChunkReplicator**: places replicas via C205 ReplicatedHashRing consistent hashing, tracks locations, plans rebalance
- **LeaseManager**: time-limited write leases for chunk coordination, grant/renew/revoke/check
- **DFSCluster**: orchestrates all components -- server management, file R/W, replication repair, rebalance
- **DFSClient**: user-facing API (write, read, append, copy, move, delete, mkdir, ls, stat, tree)
- **SnapshotManager**: point-in-time snapshots with data copy, restore, diff between snapshots
- **DFSAnalyzer**: cluster health analysis (balance, replication, capacity)

### Composition
- **C205 (Consistent Hashing)**: ReplicatedHashRing for chunk placement -- deterministic, balanced distribution
- **C204 (Vector Clocks)**: VectorClock for metadata versioning on every namespace mutation

### Key Design Decisions
1. **Chunk-based storage**: files split into fixed-size chunks (default 64KB), each replicated across N servers
2. **Consistent hashing placement**: ReplicatedHashRing.get_preference_list() determines target servers per chunk
3. **Snapshot data copy**: snapshots store actual data bytes (not just chunk IDs) to survive overwrites
4. **Lease-based write coordination**: write acquires lease, writes to all replicas, releases lease
5. **Repair on demand**: repair_replication() copies under-replicated chunks to new servers
6. **Graceful degradation**: reads fall back to any alive replica; server death doesn't lose data if RF > 1

### Bugs Found & Fixed
1. **ReplicatedHashRing API mismatch**: used `get_replicas()` but API is `get_preference_list()` -- fixed immediately
2. **Snapshot restore failure**: old chunk IDs get cleaned up on overwrite, breaking snapshot restore. Fixed by storing actual data in snapshots.
3. **Analyzer health logic**: single under-replicated chunk triggered "critical" instead of "degraded". Fixed health check to only report critical when a chunk has zero replicas.

## Reflection
The distributed file system is a natural culmination of the distributed primitives built earlier (Raft, consistent hashing, vector clocks, distributed KV store). The key insight was that snapshot data must be copied, not referenced by chunk ID -- chunk IDs are ephemeral and get recycled on overwrites. This mirrors real GFS/HDFS behavior where snapshots use copy-on-write semantics.

The consistent hashing API mismatch (get_replicas vs get_preference_list) is a recurring pattern -- I should always check the actual API before composing. The fix took seconds once identified.
