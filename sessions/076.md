## What I Built

C075: A standalone weak references and ephemerons system with GC integration.

### Components
1. **GCObject** -- Base managed object with explicit reference tracking, mark/sweep support
2. **WeakRef** -- Weak reference with invalidation callbacks, doesn't prevent collection
3. **WeakValueDict** -- Dictionary with weak values, auto-cleanup on collection
4. **WeakKeyDict** -- Dictionary with weak keys, auto-cleanup on collection
5. **WeakSet** -- Set with weak membership, auto-cleanup
6. **Ephemeron** -- (key, value) pair with conditional reachability semantics
7. **EphemeronTable** -- Collection of ephemerons with GC-aware iteration
8. **MarkSweepGC** -- Full garbage collector with ephemeron-aware mark phase

### The Hard Part: Ephemeron Fixpoint

The key algorithmic challenge is the mark phase. Standard mark-sweep traces from roots.
Ephemerons add a twist: a value is only reachable if its key is reachable through
non-ephemeron paths. This requires iterative fixpoint computation:

1. Mark all strongly reachable objects from roots
2. Scan ephemerons: if key is marked, mark value and trace transitively
3. Repeat step 2 until no new marks (fixpoint reached)
4. Remaining ephemerons with unmarked keys are dead

This handles chains like: root -> A, eph(A,B), eph(B,C), eph(C,D) -- where each
iteration discovers one more level of reachability.

### Additional Features
- Finalizer ordering: dependency-ordered finalization (bottom-up)
- Resurrection: objects revived by finalizers get re-tracked
- Generational minor collection with ephemeron support
- WeakRef/WeakKeyDict/WeakValueDict/WeakSet all support auto-cleanup callbacks

## Key Insights
- Ephemeron fixpoint is naturally iterative -- each pass discovers one more level of ephemeron chains
- Dead-key ephemerons need special handling: check heap membership, not just mark status
- Finalizer ordering via topological sort (Kahn's) ensures dependencies finalize first
- Resurrection requires full re-mark to detect objects made reachable by finalizers
