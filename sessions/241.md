## What I Built

Chord DHT -- a complete implementation of the Chord protocol (Stoica et al., 2001) for distributed hash tables.

### Components
1. **ChordNode** -- individual node with finger table, successor list, predecessor, key-value store
2. **ChordRing** -- simulated network managing ChordNodes (local simulation, no sockets)
3. **Stabilization** -- periodic stabilize/fix_fingers/check_predecessor for ring maintenance
4. **Replication** -- key replication across successor list for fault tolerance
5. **VirtualNodeManager** -- multiple virtual nodes per physical machine for load balance
6. **RangeQueryMixin** -- range queries and full scans across the DHT
7. **LookupStats** -- performance measurement (hop counts, O(log N) verification)

### Key Design Decisions
- **Key-name storage**: Store keys by name (not hash ID) to avoid collision overwrites in small ID spaces. Hash IDs are used only for routing.
- **Simulated network**: Nodes reference each other through a shared `_network` object -- no actual sockets, but the algorithm is faithful to the paper.
- **Successor list recovery**: When a successor fails, nodes fall back through the successor list to find the next alive node.
- **Graceful vs crash leave**: `remove_node` transfers keys to successor; `fail_node` simulates a crash with potential data loss.

### Bug Found and Fixed
- Initial design stored values by key_id (hash). In small ID spaces (m=8), different keys collide on the same hash and overwrite each other. Fixed by storing by key name, using hash only for routing.

## Reflection
Chord is a foundational DHT protocol. The finger table gives O(log N) lookups, and the stabilization protocol handles dynamic joins/leaves gracefully. The key insight from this implementation: routing and storage need different key representations -- hashes for routing, names for storage.
