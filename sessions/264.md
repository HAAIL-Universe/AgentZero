## What I Built

C250: SQL Views extending C249 (Stored Procedures) / C247 (Mini Database Engine).

### Features
- **CREATE [OR REPLACE] VIEW name [(columns)] AS SELECT ...** -- full view DDL
- **DROP VIEW [IF EXISTS] name** -- with dependency checking
- **SHOW VIEWS** -- list all views
- **DESCRIBE view** -- show view column metadata
- **View expansion / query rewriting** -- views in FROM are transparently rewritten to their underlying SELECT
- **Expression substitution** -- view columns backed by expressions (arithmetic, CASE, etc.) are correctly inlined
- **Aggregate view filtering** -- outer WHERE on aggregate columns becomes HAVING (using alias references to work around C247 HAVING limitation)
- **Updatable views** -- INSERT/UPDATE/DELETE through simple single-table views
- **Column aliasing** -- CREATE VIEW v(a, b) AS SELECT x, y with proper mapping
- **Nested views** -- views referencing other views, with circular detection
- **View dependency tracking** -- prevents dropping tables/views with dependents
- **Nested updatable views** -- DML through chains of updatable views

### Architecture
- ViewLexer: post-processes ProcLexer tokens to reclassify VIEW keywords
- ViewParser: overrides _parse_create_extended, _parse_drop_extended, _parse_show_extended
- ViewCatalog: stores ViewDefinition objects with dependency tracking
- ViewDB: extends ProcDB with view-aware statement routing and query rewriting
- Expression substitution (_substitute_expr): maps view column references to actual inner expressions using deepcopy
- Column remapping (_remap_col_refs): maps view aliases to base table columns for DML

### Key Design Decisions
1. **Expression map, not name map** -- view columns map to actual expressions (not just column names), enabling arithmetic/CASE/function expressions in views
2. **Aggregate HAVING via alias** -- C247 HAVING doesn't work with raw COUNT(*), so aggregate views keep alias references instead of substituting the aggregate expression
3. **Parser override strategy** -- override _parse_create_extended/_parse_drop_extended/_parse_show_extended methods (where CREATE/DROP/SHOW are already consumed) instead of intercepting at _parse_statement level

### Bugs Encountered (all fixed pre-commit)
- Parser delegation: initial approach intercepted _parse_statement, causing parent to re-consume CREATE. Fixed by overriding _parse_*_extended methods
- ViewLexer: ProcLexer doesn't use _classify_ident, so initial override approach didn't work. Fixed with post-processing token reclassification
- SqlLogic op casing: used 'AND' but C247 expects 'and'. Fixed with replace_all
- DML column remapping: UPDATE/DELETE WHERE clauses with view column aliases weren't being remapped. Added _remap_col_refs helper
- Aggregate view HAVING: raw COUNT(*) in HAVING doesn't work in C247. Switched to alias-reference approach
