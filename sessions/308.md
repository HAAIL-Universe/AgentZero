# Session 308 -- Streaming Generation Timeout & Recovery

**Date:** 2026-03-18
**Focus:** Implement research paper: streaming_generation_timeout.md

## What I Did

Implemented all 4 phases of the Streaming Generation Timeout research paper.

### Phase 1: Per-Token Timeout for vLLM Streaming (agent_zero_inference.py)
- Replaced synchronous line-by-line SSE reading with queue-based approach
- Background daemon thread reads from vLLM, puts tokens in bounded queue (maxsize=256)
- Main thread consumes from queue with configurable per-token timeout (default 30s)
- Total generation time cap (default 300s) prevents infinite streams
- Connection-level timeout on urlopen uses same token timeout value

### Phase 2: Local Model Thread Timeout (agent_zero_inference.py)
- Changed `thread.join()` to `thread.join(timeout=stream_total_timeout)`
- Made generation thread a daemon thread (cleanup on process exit)
- Logs error if thread is still alive after timeout

### Phase 3: Error Classification (agent_zero_server.py + error_responses.py)
- Added `generation_timeout` and `generation_connection_error` codes to error_responses.py
- Server now classifies exceptions: timeout vs connection vs generic
- User-friendly messages: "I'm sorry, I'm taking longer than usual..." for timeouts
- No internal details ever leak to client (already enforced by safe_error)

### Phase 4: Malformed Chunk Logging (agent_zero_inference.py)
- Replaced silent `continue` on parse errors with `_log.warning("vllm_chunk_parse_error")`
- Parse errors for JSON decode, missing keys, and index errors all logged

### Config (config.py)
- `stream_token_timeout`: 5-120s, default 30s
- `stream_total_timeout`: 30-600s, default 300s
- Also updated non-streaming vLLM tool-calling timeout to use config instead of hardcoded 120
