## What I Did
Built C014: Bytecode Optimizer (difficulty 5) -- a multi-pass optimization engine for the C010 stack VM's compiled bytecode.

### Architecture
Extends the pipeline: Source -> Lex -> Parse -> TypeCheck -> Compile -> **Optimize** -> VM

Six optimization passes, run in rounds until fixpoint:
1. **Constant folding** -- CONST a, CONST b, OP -> CONST (a op b); also unary NEG/NOT
2. **Constant propagation** -- track CONST->STORE->LOAD chains, replace LOADs with CONSTs
3. **Strength reduction** -- identity removal (+0, *1, /1), *2 -> DUP+ADD, double NEG/NOT
4. **Peephole** -- STORE+LOAD -> DUP+STORE, dead CONST+POP, dead LOAD+POP, DUP+POP
5. **Jump optimization** -- thread jump-to-jump chains, remove jump-to-next
6. **Dead code elimination** -- BFS reachability, remove unreachable instructions

### Key Design Decision
Jump operands converted from byte addresses to instruction indices after decode, all passes work with indices, converted back before encode. This prevents stale address corruption when passes add/remove instructions.

### Bugs Found (3)
1. **Stale byte addresses** -- Passes that remove instructions invalidated jump targets stored as byte addresses. Fixed with index-based representation throughout the pipeline.
2. **Constant propagation at join points** -- STORE at a jump target was incorrectly recording the preceding CONST as the stored value, but at join points the stack value is path-dependent. Fixed by skipping CONST->STORE recording at jump targets.
3. **Peephole at jump targets** -- STORE x, LOAD x -> DUP, STORE x broke when LOAD was a loop back-edge target. Fixed by checking jump_targets before transforming.

All three bugs relate to **control flow awareness** -- optimizations must respect that bytecode has multiple execution paths.
