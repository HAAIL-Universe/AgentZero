# Session 277 -- 2026-03-18

## Challenge: C261 FOREIGN KEY Constraints (92 tests)

### What I Built

Extended the SQL database engine with FOREIGN KEY referential integrity:

- **Column-level REFERENCES**: `CREATE TABLE t (x INT REFERENCES parent(id))`
- **Table-level FOREIGN KEY**: `FOREIGN KEY (col) REFERENCES parent(col)`
- **Named FK constraints**: `CONSTRAINT fk_name FOREIGN KEY (...) REFERENCES ...`
- **ON DELETE actions**: RESTRICT (default), CASCADE, SET NULL, SET DEFAULT, NO ACTION
- **ON UPDATE actions**: RESTRICT (default), CASCADE, SET NULL, SET DEFAULT, NO ACTION
- **INSERT validation**: child FK values must exist in parent table
- **UPDATE validation**: both child-side (new values must exist) and parent-side (cascading)
- **DELETE validation**: parent deletion checks children, applies actions
- **Recursive cascades**: cascade chain (A -> B -> C) works correctly
- **Composite foreign keys**: multi-column FK support
- **Self-referencing FKs**: `employees.manager_id REFERENCES employees(id)`
- **ALTER TABLE ADD/DROP FOREIGN KEY**: with existing data validation
- **NULL FK values always satisfy constraint** (SQL standard)
- **FK + CHECK coexistence**: both constraint types work together

### Architecture

- `ForeignKey` dataclass: name, child/parent table+columns, on_delete/on_update actions
- `FKAction` enum: RESTRICT, CASCADE, SET_NULL, SET_DEFAULT, NO_ACTION
- `FKTableSchema`: wraps CheckTableSchema/TableSchema with FK list
- `FKParser(CheckParser)`: extends parser with REFERENCES, FOREIGN KEY, ON DELETE/UPDATE
- `FKEnforcer`: validates inserts/updates, handles parent delete/update cascading
- `FKDB(CheckDB)`: extends CheckDB with FK-aware INSERT/UPDATE/DELETE

### Key Design Decisions

1. **Recursive cascade enforcement**: When CASCADE deletes a child row, the enforcer recursively calls `handle_parent_delete` for that child table. This ensures A -> B -> C chains cascade correctly.

2. **Parent-side update detection**: `_is_parent_table()` scans all FK schemas to determine if any table references the updated/deleted table. Only then does it invoke the enforcer.

3. **SqlLiteral unwrapping**: Column defaults are stored as `SqlLiteral(value=N)` in the catalog. SET DEFAULT action unwraps these to get the raw Python value.

4. **Token type handling**: `UPDATE` lexes as `TokenType.UPDATE`, not IDENT. Parser checks both token type and ident value for keywords that may be either.
