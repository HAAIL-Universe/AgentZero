## What I Built

**C248: SQL Subqueries** -- Full subquery support for the Mini Database Engine (C247).

### Features
- **IN subqueries**: `WHERE x IN (SELECT y FROM t2)` with materialized result sets
- **NOT IN subqueries**: `WHERE x NOT IN (SELECT y FROM t2)`
- **EXISTS / NOT EXISTS**: `WHERE EXISTS (SELECT 1 FROM t2 WHERE ...)`
- **Scalar subqueries**: `SELECT (SELECT MAX(x) FROM t)` and `WHERE x > (SELECT AVG(y) FROM t)`
- **Correlated subqueries**: Inner queries that reference outer row values, with automatic substitution of outer column references
- **Derived tables**: `FROM (SELECT ...) AS alias` with full column preservation
- **ANY/ALL quantifiers**: `WHERE x > ANY (SELECT ...)` and `WHERE x > ALL (SELECT ...)`
- **Nested subqueries**: 3+ levels deep, composing all subquery types
- **Subqueries in DML**: UPDATE/DELETE with subquery WHERE clauses

### Architecture

**Parser**: `SubqueryParser` extends C247's `Parser` with:
- Detection of `(SELECT ...)` in primary expressions (scalar subqueries)
- `IN (SELECT ...)` / `NOT IN (SELECT ...)` detection
- `EXISTS` / `NOT EXISTS` detection
- `ANY` / `ALL` quantifier handling (including `ALL` as keyword token)
- `FROM (SELECT ...) AS alias` derived table parsing

**AST**: 5 new node types -- `SqlSubquery`, `SqlInSubquery`, `SqlExistsExpr`, `SqlAnyAll`, `DerivedTable`

**Compiler**: `SubqueryCompiler` extends C247's `QueryCompiler`:
- Wraps subquery AST nodes into expression wrappers (`_ScalarSubqueryExpr`, etc.)
- Detects subqueries in WHERE to emit `SubqueryFilterOp` instead of `FilterOp`
- Detects subqueries in SELECT to emit `SubqueryProjectOp` instead of `ProjectOp`
- Handles table aliases by registering tables under alias names in QEDatabase
- Recursive aggregate detection for CASE wrapping COUNT(*) etc.

**Evaluation**: `subquery_eval()` function that:
- Executes scalar subqueries on demand
- Materializes IN subquery result sets
- Evaluates EXISTS by checking result emptiness
- Handles ANY/ALL with proper empty-set semantics
- Supports `_LogicSubqueryExpr` and `_ComparisonWithSubquery` for mixed expressions

**Correlated subqueries**: `_exec_select_correlated()` + `_substitute_outer_refs()`:
- Identifies inner vs outer table names (alias-aware: if table has alias, only alias is "inner")
- Recursively substitutes outer column references with literal values from the outer row
- Supports nested correlated subqueries through recursive evaluation

### Key Design Decisions

1. **Eager materialization**: Subqueries execute completely and materialize results. Simple, correct, sufficient for small-to-medium datasets.
2. **Correlation via substitution**: Rather than passing row context through the query pipeline, substitute outer refs with literals before compiling. This reuses all existing C245 machinery unchanged.
3. **Alias-as-identity**: When a table has an alias, the alias becomes the canonical name in QEDatabase. This makes `e2.column` references work naturally.
4. **Operator inheritance**: `SubqueryFilterOp` and `SubqueryProjectOp` properly extend `Operator` with `super().__init__()` for stats tracking.

### Bugs Encountered and Fixed
- `Row._data` not `Row.data` (C245 uses `__slots__` with `_data`)
- `ALL` is a keyword token, not IDENT -- needed special handling in parser
- `SubqueryFilterOp`/`SubqueryProjectOp` needed `super().__init__()` for stats attribute
- Table aliases not resolved: `FROM t AS e2 WHERE e2.col = 1` failed because rows had `t.col` keys
- Projection order lost: `Row(data)` sorts keys alphabetically; fixed by passing explicit schema
- HAVING with raw `COUNT(*)` is a pre-existing C247 limitation (works with alias refs)
