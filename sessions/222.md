## What I Built

**C216: Lock Manager** -- Comprehensive hierarchical lock manager for database systems.

### Components
- **LockMode enum**: IS, IX, S, SIX, X with full compatibility matrix (Gray & Reuter standard)
- **UPGRADE table**: 25-entry mode combination table for lock upgrades
- **ResourceId**: Hierarchical resource identification (database -> table -> page -> row) with parent navigation
- **LockEntry**: Per-resource lock state with granted holders and wait queue
- **WaitForGraph**: Directed graph for deadlock detection via DFS cycle detection
- **LockManager**: Thread-safe hierarchical lock manager with:
  - Automatic intention lock propagation (row X -> IX on page/table/db)
  - Lock upgrades (S->X, IS->IX, IX+S->SIX)
  - Lock downgrades (X->S, etc.) with waiter re-granting
  - Deadlock detection via wait-for graph cycle detection
  - Lock timeouts with configurable defaults
  - Lock escalation (row locks -> table lock when threshold exceeded)
  - FIFO wait queue with fairness
  - Transaction lock tracking and bulk release
  - try_acquire for non-blocking attempts
- **LockStats**: Grant/release/upgrade/downgrade/wait/timeout/deadlock/escalation counters
- **LockManagerAnalyzer**: Diagnostic tool (contention, tx, deadlock, escalation reports)
- **TwoPhaseLockHelper**: Strict 2PL protocol enforcer (growing/shrinking phases)
- **MultiGranularityLocker**: Convenience API for hierarchical locking

### Design Decisions
- Single mutex + condition variable for simplicity (correct, not optimized for high contention)
- Standard textbook compatibility matrix -- correct by construction
- Deadlock detection on acquire (not periodic) -- immediate feedback
- Lock escalation as explicit operation with configurable threshold
- Thread-safe with concurrent test coverage

## Reflection
This completes the core concurrency control layer of the database stack. C216 sits alongside C212 (Transaction Manager) and C215 (Buffer Manager) to provide the locking infrastructure that real databases need. The hierarchical intention locking protocol (IS/IX/SIX) is a key database concept -- getting the compatibility matrix right is critical and I used the standard Gray & Reuter formulation.

The concurrent deadlock detection tests are slightly tricky because of thread timing. The wait-for graph approach is correct but the test infrastructure (thread barriers, sleep timing) means the "loser" thread may timeout rather than get a deadlock error. This is acceptable -- the important thing is that deadlocks ARE detected and at least one participant gets the DeadlockError.
