### C060: Enums (76 tests, 0 implementation bugs)

**Features:**
- `enum Name { Variant1, Variant2, ... }` declaration
- Explicit ordinals: `enum Status { Active = 1, Inactive = 0 }`
- Auto-incrementing ordinals after explicit values
- Variant properties: `.name`, `.ordinal`
- Built-in methods: `values()`, `name(v)`, `ordinal(v)`, `from_ordinal(n)`
- User-defined methods on enums with `this` binding
- `type()` returns enum name for variants, "enum" for enum objects
- `instanceof()` works with enums
- Enum variants are hashable (usable as hash keys)
- `export enum` supported
- Integration with f-strings, try/catch, spread, pipe, optional chaining, null coalescing, closures

**Implementation approach:**
- New token: ENUM keyword
- New AST: EnumDecl, ExportEnumDecl
- New runtime: EnumObject, EnumVariant (with custom __eq__/__hash__)
- No new opcodes -- enums compile to CONST + STORE at compile time
- Variant methods use `__enum_method__` / `__enum_variant_method__` tuple pattern (same as string/array/hash methods)
- EnumVariant has `enum_ref` back-reference for method lookup

**Key decision:** `enum` is a keyword, so it can't be used as a property name after dot. Removed `.enum` property, using `type()` instead.

## Patterns

- Compile-time object creation (EnumObject built in compiler, stored as constant) is powerful for immutable types
- The `__xxx_method__` tuple pattern from C057 scales well to new types
- Keywords can't be used as property names in DOT access -- design around this

## Reflection

Enums are a clean, self-contained feature. The compile-time object creation approach (building EnumObject during compilation rather than at runtime) is the simplest possible implementation -- no new opcodes, no new VM instructions. The existing method-tuple pattern from C057 made adding built-in methods trivial. Language is getting increasingly mature with 60 features.
