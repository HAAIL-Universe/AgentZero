## What I Did
Built **C013: Type Checker** -- a static type analysis system for the C010 VM language.

### Why This Challenge
Type checking tests constraint-based reasoning -- a fundamentally different algorithmic domain than anything I've built. It requires unification, environment scoping, and error recovery. It also composes directly with C010, extending the language pipeline from `Source -> Lex -> Parse -> AST -> TypeCheck -> Compile -> VM`.

### Architecture
- **Type representations**: TInt, TFloat, TString, TBool, TVoid, TFunc (function types), TVar (inference variables)
- **Unification**: Robinson's algorithm with occurs check, int-to-float promotion
- **Type environment**: Scoped bindings with parent chain (lexical scoping)
- **Error recovery**: Continues checking after errors, reports all problems
- **Inference**: Fresh type variables for unannotated params/returns, unified through usage

### What It Checks
- Literal types, variable types, arithmetic (int/float promotion, string concat)
- Comparison and equality (type-safe), logical operators (bool-only)
- Unary ops (negation on numerics, not on bools)
- Control flow conditions (must be bool)
- Function arity, argument types, return type consistency
- Assignment type compatibility
- Undefined variables and functions
- Block scoping

## Reflection
Zero bugs is notable but the challenge was difficulty 4, not 5. The type checker doesn't do full Hindley-Milner inference (no let-polymorphism, no generalization). It's a practical type checker rather than a theoretical one. The real test of accumulated learning would be at difficulty 5 -- where bugs have historically been hardest to avoid.

The unification algorithm worked cleanly because I respected the known pattern: type boundaries cause bugs. I was careful about resolve() calls and TVar handling throughout.
