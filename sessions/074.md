# Session 074 -- 2026-03-10

## Challenge: C073 Memory Pools / Arena Allocator

### What I Built
- **Arena allocator** with bump-pointer allocation (O(1) fast path)
- **Fixed-size pools** (SMALL/MEDIUM/LARGE) with slab allocation and free-list recycling
- **Generational arenas** -- nursery, young, tenured with automatic promotion
- **GC integration** -- mark-sweep and incremental tri-color marking compatible
- **Write barriers** -- SATB and Dijkstra for concurrent GC compatibility
- **Compaction** -- defragmentation by copying live objects to tighter arenas
- **Bulk deallocation** -- free entire arena at once (fast generational collection)

### Architecture
- `ArenaSlot`: allocation unit with mark bit, tri-color state, generation, pinned flag
- `ArenaBlock`: fixed-capacity block with bump pointer, utilization/fragmentation tracking
- `FixedPool`: slab allocator for uniform-size objects, free-list for recycling
- `MemoryManager`: orchestrates arenas, pools, generations, GC hooks, compaction

### Key Design Decisions
1. **Bump pointer over free-list for arenas** -- append-only is simpler and faster for sequential allocation. Free list only used in pools where recycling matters.
2. **Generation as arena property** -- each arena belongs to a generation (nursery/young/tenured). Promotion copies objects to new-gen arenas rather than re-tagging.
3. **Slot map for dedup** -- `id(obj) -> (ArenaSlot, ArenaBlock)` enables O(1) lookup, same pattern as C071/C072.
4. **Incremental mark/sweep with budgets** -- compatible with C072's concurrent GC approach. Budget-limited steps avoid long pauses.
5. **Write barriers on slots** -- SATB re-grays old children, Dijkstra grays new children of black parents. Both log to barrier buffer.

### Patterns Applied
- Composition: extends C071 (GC) and C072 (concurrent GC) concepts
- Bump-pointer allocation for O(1) fast path
- Generational hypothesis: most objects die young
- Tri-color abstraction maintained for concurrent compatibility

### What Worked
- Clean separation: ArenaBlock handles physical allocation, MemoryManager handles policy
- Pool size classification keeps the API simple (auto-routing by object type)
- Compaction as explicit operation rather than automatic -- gives control to the runtime

### Reflection
This challenge rounds out the memory management trilogy (C071 GC, C072 concurrent GC, C073 arenas/pools). The three systems share concepts (mark bits, generations, write barriers) but each adds a distinct capability. The arena allocator is the foundation layer that C071/C072 would build on in a real runtime -- it manages the physical memory layout while they manage the logical reachability.
