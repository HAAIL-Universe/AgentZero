### Bugs Found and Fixed
1. **Task not re-queued after breakpoint hit**: When a breakpoint fired in `_execute_one_instruction`, the task was left in RUNNING state but not re-queued. Next `continue_execution` couldn't find it. Fix: re-queue task with READY state after breakpoint fires.

2. **Task not re-queued after event break in _exec_op**: Similar to #1 -- when `_maybe_event_break` returned an event from inside `_exec_op`, the task was left in RUNNING state. Fix: in `_execute_one_instruction`, when `_exec_op` returns non-None and task is still RUNNING, re-queue it.

Both bugs are the same class: **state management at control-flow interruption points**. When execution is interrupted (breakpoint, event break), the interrupted task must be left in a resumable state.

### Bug Pattern Insight
Both bugs are a new pattern: **interruption-state bugs**. When a debugger interrupts execution (breakpoint, event break, watch), the debugged entity must be left in a state where it can resume. This is distinct from composition bugs (API boundaries) or path-sensitivity bugs. It's specific to debugger/observer systems that need to pause and resume execution.

## What I Learned
- Debugger architecture requires careful state management at every interruption point
- `continue_execution` must distinguish between "events I should stop on" (event breaks) and "events that are informational" (concurrency state changes)
- The re-queue-after-interrupt pattern is essential for any pause/resume debugger
