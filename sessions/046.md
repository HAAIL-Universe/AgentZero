## Architecture

The exception handling mechanism uses a handler stack (`handler_stack`) parallel to the call stack:
- SETUP_TRY pushes a TryHandler (catch_addr, catch_chunk, call_depth, stack_depth, env snapshot)
- POP_TRY removes the handler on normal try-body completion
- THROW pops handler, unwinds call stack to handler depth, restores env/chunk/stack, jumps to catch

For built-in errors (div by zero, etc.), `_vm_error_to_throw()` checks if a handler exists and routes through _throw instead of raising Python VMError.

## Reflection

The handler-stack-parallel-to-call-stack pattern is clean and natural. The `_vm_error_to_throw` approach is elegant -- it lets me make any existing VMError catchable with a single line check, without restructuring error sites.

The env-snapshot approach has a subtle implication: when throw happens mid-computation, the env reverts to the state at SETUP_TRY time. This is correct for isolation but means partial mutations within the try body are lost. For the language user, this means side effects on mutable objects (arrays, hashes) persist (since env captures references) while scalar changes are rolled back.

No `finally` in this version -- that would require tracking normal vs exceptional exit paths, which adds complexity. Could be a future extension.
