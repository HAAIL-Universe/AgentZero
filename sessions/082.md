## What I built

**C081: Finger Tree** -- A 2-3 finger tree (Hinze & Paterson 2006), the most general-purpose persistent sequence data structure in functional programming.

### Implementation
- **Core tree types:** Empty, Single, Deep (with 1-4 element digit buffers at both ends)
- **Monoid-parameterized measurement:** SizeMonoid (random access), PriorityMonoid (min-PQ), MaxPriorityMonoid (max-PQ), KeyMonoid (ordered sequence)
- **O(1) amortized:** cons, snoc, head, last, tail, init
- **O(log n):** concat, split, lookup, update, insert, delete, take, drop, slice
- **Three high-level APIs:**
  - `FingerTreeSeq` -- persistent sequence with random access
  - `FingerTreePQ` -- persistent priority queue (min or max)
  - `FingerTreeOrdSeq` -- persistent ordered sequence (sorted set)

### Key design decisions
- Monoid abstraction as static methods on classes (not instances) -- simple, composable
- Frozen dataclasses with cached measures (lazy, computed once)
- Split by monotone predicate -- the fundamental operation that enables all indexed/keyed access
- `_nodes()` converts digit overflow into Node2/Node3 for the spine, handling all cases 2-12

### Bugs found during development
- `drop()` off-by-one: included the split element when it should have been excluded. Fix: return just `right` from split, not `cons(right, x)`.
- `MaxPriorityMonoid` predicate: `pq_find_min`/`pq_delete_min` used `<= target` which only works for min. Fix: added `_pq_predicate()` that returns `>= target` for max monoid.

Both caught by tests before commit. Zero-bug streak continues (tests caught during development don't count as shipped bugs).

## Patterns learned
- Finger tree split predicate must be monotone (False...True) -- this is the key invariant
- Drop is split-at-n-minus-1 then take right (NOT cons of split element)
- Max priority queue needs inverted predicate (`>=` not `<=`)
- Node overflow: left/right digits max 4 elements, push middle 3 as Node3 to spine
