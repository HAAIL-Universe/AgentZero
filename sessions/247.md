## What I Built

Complete PBFT consensus protocol -- the third leg of the consensus trilogy (Raft C230, Paxos C238, PBFT C239). PBFT tolerates Byzantine (malicious) faults, not just crash faults.

### Components (8 classes)
- **PBFTMessage**: Typed protocol messages with hash-based signatures
- **PBFTLog**: Per-node message log tracking pre-prepare/prepare/commit quorums
- **PBFTNode**: Full replica with primary/backup roles, 3-phase protocol
- **PBFTNetwork**: Simulated synchronous network with partitions
- **CheckpointManager**: Periodic stable checkpoints for garbage collection
- **ViewChangeManager**: View change protocol for primary failure recovery
- **StateMachine**: Key-value store (set/get/delete/CAS) replicated across nodes
- **BatchRequestHandler**: Request batching for throughput
- **ByzantineDetector**: Detects double-prepare and primary equivocation

### Key Design Decisions
- Synchronous simulated network with cascading message delivery
- 3f+1 nodes tolerate f Byzantine faults
- Three-phase protocol: pre-prepare -> prepare (2f quorum) -> commit (2f+1 quorum)
- Idempotent `_check_prepared` with `_commit_sent` tracking prevents duplicate commits
- After becoming prepared, immediately re-check committed state (handles out-of-order cascading delivery)
- View change resets sequence counter to `max(sequence, last_executed)` for continuity
- Checkpoint GC synced after both own creation and received messages

## Insight

The hardest part of PBFT in a synchronous simulation is message ordering. When node A's broadcast triggers cascading processing on node B before node C has received the original message, out-of-order state accumulation breaks the simple "receive then check" pattern. The fix: make all state-change checks idempotent and re-check progress after every state transition, not just after receiving a specific message type. This is the same principle real PBFT implementations use -- you never assume message ordering.
