## What I Did

### 1. A2 Verification Findings (2 fixes)

A2 flagged two issues from Session 299 verification:

- **Exception leak in /api/load-model (line 1431):** Raw `inference.load_error` was returned directly to client, potentially exposing internal paths/errors. Fixed: now uses `safe_http_error()` which returns generic message and logs the real error server-side.

- **Missing per-user rate limit in /ws/voice:** The /ws/chat handler had both per-connection (`ws_rate.check()`) AND per-user API rate limit (`rate_limiter.check()`), but /ws/voice only had per-connection. Voice users could bypass the per-user rate limit. Fixed: added identical per-user rate limit block to /ws/voice.

### 2. Async Event Loop Safety (research paper implemented)

Eliminated all 11 deprecated `asyncio.get_event_loop()` calls across 3 files:

- **agent_zero_server.py (8 sites):** Replaced `loop = asyncio.get_event_loop(); await loop.run_in_executor(None, fn)` with `await asyncio.to_thread(fn)`. For `.time()` calls, replaced with `asyncio.get_running_loop()`.

- **cognitive_runtime.py (2 sites):** Same pattern -- `to_thread()` replacement.

- **tool_runtime.py (2 sites):** Converted blocking `subprocess.run()` (10-second timeout that froze entire event loop) to `asyncio.create_subprocess_exec()` with `asyncio.wait_for()` for non-blocking execution.

Why this matters: `get_event_loop()` is deprecated in Python 3.10+ and emits DeprecationWarning in 3.12. The blocking subprocess calls froze ALL WebSocket connections for up to 10 seconds during MQ operations.

### 3. Silent Exception Data Loss (research paper partially implemented)

- **behavioural_shadow.py:** Replaced two `except Exception: pass` blocks in commitment tracking with `logger.warning()` calls. Previously, if `create_commitment` or `db_update_commitment_status` raised (network timeout, constraint violation), the error was silently swallowed -- user's commitment acknowledged but never recorded.

- **clear_user_data re-auth guard:** Added `ClearDataRequest(password: str)` model. DELETE /api/user/clear-data now requires password re-confirmation before proceeding. Returns 403 on wrong password. Adds audit log entry. Previously, a stolen JWT alone could permanently delete all user data.

- **Soft-delete deferred:** The research paper also proposes a 48-hour soft-delete window with DB schema changes. That's a larger change requiring migration scripts -- left for a future session.

## Key Decisions

- Left urllib calls in voice.py and agent_zero_inference.py as-is -- they're sync functions already properly wrapped with `to_thread()`/`run_in_executor()` at call sites
- Used confirmation-by-password for destructive ops rather than confirmation-by-string ("DELETE") -- stronger security
- Deferred soft-delete to avoid risky DB migration in a single session
