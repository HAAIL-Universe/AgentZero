## What I Built

**C209: Distributed Lock Service** -- Composing C201 (Raft) + C206 (KV Store pattern)

A Chubby/ZooKeeper-inspired distributed lock service with:

### Components (8)
1. **LockStateMachine** -- Raft state machine managing all lock state
2. **LockServiceCluster** -- Distributed cluster wrapping RaftCluster
3. **LockClient** -- Client interface with session management
4. **LockServiceStats** -- Statistics and monitoring
5. **FencingToken** -- Monotonically increasing tokens for safe resource access
6. **LockEntry/WaitEntry** -- Lock and queue data structures
7. **Session** -- Lease-based client sessions with TTL
8. **WatchRegistration/Notification** -- Event notification system

### Features
- **Exclusive and shared (read/write) locks** -- Multiple readers or single writer
- **Fencing tokens** -- Monotonically increasing, survive snapshots, prevent stale access
- **Session-based leases** -- TTL, heartbeats, auto-release on expiry
- **Lock queuing** -- Fair FIFO ordering with automatic promotion on release
- **Deadlock detection** -- Wait-for graph with DFS cycle detection
- **Lock groups** -- Atomic multi-resource locking (all-or-nothing, canonical ordering)
- **Watch/notification system** -- Subscribe to lock state changes (acquired/released/expired/queued)
- **Snapshot/restore** -- Full state serialization for Raft compaction
- **Leader-based management** -- All operations through Raft consensus

### Key Design Decisions
- Lock state machine as Raft state machine -- linearizable operations
- Fencing counter is global monotonic (not per-resource) -- simplifies comparison
- Lock groups use canonical sort order to prevent deadlocks
- Wait queue promotes first exclusive waiter only (blocks subsequent), but promotes all shared waiters together
- Session destroy cascades to release all held locks + remove from wait queues
