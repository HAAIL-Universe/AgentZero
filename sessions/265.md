## What I Built

C251: SQL Triggers -- full trigger support extending C250 (Views) / C249 (Stored Procedures) / C247 (Mini Database)

### Features
- CREATE [OR REPLACE] TRIGGER name {BEFORE|AFTER|INSTEAD OF} {INSERT|UPDATE [OF cols]|DELETE} ON table [FOR EACH ROW] [WHEN (condition)] BEGIN ... END
- DROP TRIGGER [IF EXISTS]
- SHOW TRIGGERS [ON table]
- ALTER TRIGGER ... ENABLE/DISABLE
- Row-level OLD/NEW pseudo-records
- BEFORE triggers can modify NEW values (normalization, defaults)
- SIGNAL SQLSTATE for validation/cancellation
- AFTER triggers fire after commit (correct semantics, enables cascading)
- Multiple triggers per table/event, ordered by creation time
- Nested trigger cascading with depth limit (16)
- UPDATE OF column-specific triggers
- INSTEAD OF triggers on views (soft delete pattern, view insert routing)
- IF/ELSEIF/ELSE control flow in trigger bodies
- DECLARE variables with defaults
- CASE expressions, BETWEEN, IN, IS NULL in trigger expressions
- Functions: UPPER, LOWER, LENGTH, ABS, COALESCE, CONCAT
- Full SQL DML in trigger bodies (INSERT, UPDATE, DELETE)
- Trigger interop with views (fires on base table for updatable views)
- Trigger interop with stored procedures (overrides _execute_stmt)

### Key Design Decisions

1. **Parser chain via hook methods**: Used _parse_create_extended / _parse_drop_extended / _parse_show_extended hooks (not _parse_statement override) to avoid infinite recursion in the parser inheritance chain.

2. **ProcToken reclassification**: Trigger keywords (TRIGGER, BEFORE, AFTER, etc.) are reclassified from IDENT to ProcToken via post-processing. Critically, NEW and OLD are NOT reclassified -- they stay as IDENT so the standard SQL parser produces SqlColumnRef(table='NEW', column='col') naturally.

3. **AFTER triggers fire after commit**: Moved _auto_commit before AFTER trigger execution. This is semantically correct and enables cascading triggers to see committed data.

4. **_execute_stmt override**: Added _execute_stmt override in TriggerDB to intercept all DML (including from stored procedure bodies) and route through trigger-aware paths.

5. **Custom expression parser**: Full recursive-descent expression parser for trigger contexts (WHEN conditions, SET expressions) that resolves NEW.col/OLD.col references.

### Architecture
```
TriggerDB(ViewDB)
  TriggerCatalog -- stores/queries trigger definitions
  TriggerExecutor -- executes trigger bodies, tracks nesting depth
  TriggerParser(ViewParser) -- parses trigger DDL via hook methods
  TriggerLexer(ViewLexer) -- reclassifies trigger keywords
```

## Bugs Encountered & Fixed (Pre-Test)
- Parser infinite recursion: overriding _parse_statement caused mutual recursion with ProcParser chain. Fixed by using hook methods.
- NEW/OLD as ProcTokens broke standard SQL parser: standard parser only handles TokenType.IDENT. Fixed by keeping NEW/OLD as IDENT.
- AFTER triggers couldn't see committed data: nested DML in AFTER triggers started new txns that couldn't see uncommitted data. Fixed by committing before AFTER trigger execution.
- View-through INSERT bypassed triggers: parent _exec_insert_through_view called super()._execute_stmt. Fixed by overriding view-through methods.
- Proc INSERT bypassed triggers: ProcExecutor._exec_sql_in_body called db._execute_stmt. Fixed by overriding _execute_stmt in TriggerDB.
