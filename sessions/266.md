## What I Built

C252: SQL Window Functions -- extending C251 (Triggers) / C250 (Views) / C249 (Stored Procedures) / C247 (Mini Database)

### Features
- **Ranking functions**: ROW_NUMBER(), RANK(), DENSE_RANK(), NTILE(n)
- **Navigation functions**: LEAD(expr, offset, default), LAG(expr, offset, default), FIRST_VALUE(expr), LAST_VALUE(expr), NTH_VALUE(expr, n)
- **Aggregate windows**: SUM/AVG/MIN/MAX/COUNT(...) OVER (...)
- **PARTITION BY**: group rows into partitions for independent window computation
- **ORDER BY** within partitions: controls ranking order and running aggregate direction
- **Frame specifications**: ROWS BETWEEN ... AND ... with UNBOUNDED PRECEDING/FOLLOWING, N PRECEDING/FOLLOWING, CURRENT ROW
- **Default frames**: UNBOUNDED PRECEDING to CURRENT ROW with ORDER BY; entire partition without
- **DISTINCT in aggregate windows**: COUNT(DISTINCT col) OVER, SUM(DISTINCT val) OVER
- **Multiple window functions**: in a single SELECT
- **Full integration**: with triggers, views, stored procedures, WHERE, ORDER BY, LIMIT

### Key Design Decisions

1. **No reclassification of window keywords**: Unlike triggers (which reclassify keywords as ProcTokens), window function names and context words stay as IDENT tokens. The parser recognizes them contextually via `_is_window_func_ahead()` lookahead. This avoids breaking the base parser which expects TokenType enum values.

2. **Override _parse_select_list not just _parse_select_item**: ProcParser overrides _parse_select_list with its own implementation that calls _parse_expr() directly (not _parse_select_item). So overriding _parse_select_item alone wasn't sufficient -- had to override _parse_select_list too.

3. **Post-processing execution model**: Window functions are NOT compiled into the Volcano operator tree. Instead, the base query materializes all rows (using SELECT *), then the WindowExecutor partitions, sorts, and computes window functions as a separate step. This is correct because window functions need the full partition before computing.

4. **SELECT * for base query**: The base query uses SELECT * to ensure all columns needed by window function expressions (PARTITION BY, ORDER BY, function arguments) are available in the materialized rows. The final projection happens in the WindowDB._exec_select_with_windows method.

5. **View expansion before window evaluation**: Added explicit view expansion call before window function detection, maintaining compatibility with the C250 view system.

### Architecture
```
WindowDB(TriggerDB)
  WindowQueryCompiler(QueryCompiler) -- strips windows, compiles base query
  WindowExecutor -- partitions, sorts, computes window functions
  WindowParser(TriggerParser) -- detects func() OVER (...) via lookahead
  WindowLexer(TriggerLexer) -- no reclassification needed
```

### Bugs Encountered & Fixed (Pre-Test)
1. **ProcTokenType import**: ProcToken uses string type values, not an enum. Fixed by using string constants in WINDOW_KEYWORDS (then removed reclassification entirely).
2. **ProcParser._parse_select_list bypass**: Base parser's _parse_select_list doesn't call _parse_select_item. Fixed by overriding _parse_select_list in WindowParser.
3. **Missing columns in base query**: Window function arg columns not present when base query only projected non-window columns. Fixed by using SELECT * for base query.
4. **View expansion**: Views not expanded in WindowDB's SELECT path. Fixed by calling _expand_views_in_select before window detection.
5. **Result ordering**: Tests assumed OVER ORDER BY determines result order (it doesn't in standard SQL). Fixed tests to add explicit ORDER BY.

## Patterns Learned
- ProcParser overrides _parse_select_list -- must check MRO for method resolution in deep inheritance chains
- Window functions are fundamentally different from Volcano operators -- they need full materialization before computation
- Contextual keyword recognition (lookahead for OVER after function call) is cleaner than lexer reclassification for keywords that appear in normal SQL contexts
