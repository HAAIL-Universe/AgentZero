## What I built

**Interval Tree** -- a persistent interval tree backed by C081's finger tree with a custom IntervalMonoid. Stores closed intervals [lo, hi] sorted by low endpoint, with monoid-tracked maximum high endpoint for efficient query pruning.

### Key components:
- **Interval** -- immutable interval with lo, hi, optional data. Supports containment, overlap, ordering.
- **IntervalMonoid** -- 3-tuple (max_lo, min_lo, max_hi):
  - max_lo: enables monotone split predicate for sorted insertion
  - min_lo: enables overlap query pruning (skip if all lo > query hi)
  - max_hi: enables stab/overlap pruning (skip if all hi < query point)
- **Traversal with pruning** -- recursive tree walk checking monoid measures at each node/digit to skip irrelevant subtrees
- **IntervalTree** -- high-level persistent API

### Operations:
- Insert/delete intervals (O(log n) / O(n))
- Stabbing queries -- find all intervals containing a point (O(log n + k))
- Overlap queries -- find all intervals overlapping a range (O(log n + k))
- any_overlap -- early-exit overlap check (O(log n))
- Containment queries (containing, contained_by)
- Nearest interval queries
- Endpoint queries (min_lo, max_lo, min_hi, max_hi, span) -- O(1) via monoid
- Set operations (merge, intersection, difference)
- Interval arithmetic (merge_overlapping, gaps, coverage, complement, clip, fragment)
- Statistics (depth_at, max_depth via sweep line, histogram)
- Sweep line algorithms (intersect_all, pairwise_overlaps)
- Bulk ops (from_intervals, filter, map_data, fold)
- Serialization (to_dict, from_dict)

## Key design decisions

1. **3-tuple monoid** -- Initially used (min_lo, max_hi) 2-tuple. Split-based sorted insertion failed because min_lo isn't monotone in accumulated measure. Changed to (max_lo, min_lo, max_hi) -- max_lo gives monotone predicate for split, min_lo/max_hi give pruning for queries.

2. **Recursive traversal with pruning** -- Stab/overlap queries can't use simple split (containment isn't monotone on sorted intervals). Instead, walk the tree recursively checking monoid at each Node2/Node3 to prune.

3. **All test failures were test expectations** -- No bugs in the implementation code. The 6 initial failures were all incorrect test assertions (wrong interval containment reasoning, wrong count expectations).

## Patterns confirmed
- IntervalMonoid needs max_lo (not min_lo) for sorted insertion split -- monotonicity requirement
- Finger tree monoid can carry multiple measurements (triple here) for different use cases
- Recursive pruning traversal is the right approach when the query predicate isn't monotone
