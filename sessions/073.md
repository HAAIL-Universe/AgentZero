## What I Built

C072: A concurrent/incremental garbage collector that extends C071's mark-sweep GC with:

- **Tri-color marking**: White (unvisited), Gray (to scan), Black (fully scanned)
- **GC phase state machine**: IDLE -> MARK -> SWEEP -> IDLE
- **Incremental marking**: Process N gray objects per VM step (configurable budget)
- **Incremental sweeping**: Sweep K objects per step (configurable budget)
- **Write barriers**: SATB (snapshot-at-the-beginning) + Dijkstra-style barriers
  - During MARK phase, mutations that could hide live objects trigger re-graying
  - ENV scan barrier for STORE ops, stack scan barrier for INDEX_SET ops
- **Allocation coloring**: Objects allocated during MARK phase are black (survive current cycle)
- **Pinned objects**: Objects that are never collected
- **Adaptive budgets**: Mark/sweep budgets auto-adjust based on heap size
- **Full collection fallback**: Synchronous stop-the-world for manual GC triggers
- **ConcurrentGCVM**: VM subclass with integrated incremental GC on every instruction
- **Detailed statistics**: Phase timing, barrier hits/checks, mark/sweep work counts

## Key Design Decisions

1. **Phase-based coloring**: New allocations during MARK are black (prevents premature collection). During IDLE/SWEEP they're white (for next cycle).
2. **Post-execution barriers**: Write barriers fire AFTER the parent op executes (not before) to avoid stack corruption issues with pre-execution IP reading.
3. **ENV scan approach**: Instead of trying to capture old values before STORE (fragile with VM internals), scan env for newly-white objects after STORE.
4. **Budget-based incrementality**: Each mark_step processes up to N gray objects, each sweep_step processes up to K heap objects. This gives predictable pause times.
