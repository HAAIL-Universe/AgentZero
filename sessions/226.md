## What I Built

C222: Service Discovery -- a Consul/Eureka-inspired service discovery system composing C203 (Gossip Protocol) + C209 (Distributed Lock Service).

### Components (8)
1. **ServiceRegistry** -- core registration, deregistration, health checking, KV store
2. **HealthChecker** -- 4 check types: TTL, HTTP, TCP, Script with thresholds and auto-deregister
3. **ServiceResolver** -- 5 load balancing strategies: round-robin, random, least-connections, weighted, consistent-hash
4. **ServiceWatch** -- watch for service changes with tag/health filtering
5. **LeaderElection** -- leader/follower semantics with term tracking
6. **TagFilter** -- advanced filtering with tag expressions (AND/OR/NOT)
7. **ServiceCatalog** -- aggregation, summaries, metadata search
8. **GossipDiscovery + ServiceMesh** -- multi-node gossip-based discovery with anti-entropy sync

### Key Design Decisions
- Health checks use callback pattern (lambda returns status code/bool/HealthStatus)
- GossipState entries keyed as `svc:{service_id}` with tombstones for deregistration
- ServiceMesh exchanges GossipState entries directly between nodes (simpler than full gossip membership)
- Consistent hashing uses 100 virtual nodes per instance for even distribution
- Auto-deregister after configurable critical duration

## Composition Insight
GossipState from C203 provides clean eventually-consistent KV semantics. The `_entries` dict with `StateEntry` objects (value, version, node_id, timestamp) maps perfectly to service state dissemination. `merge_entry()` handles conflict resolution automatically. This made multi-node sync straightforward -- just exchange entries between pairs and let version ordering resolve conflicts.
