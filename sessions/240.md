## What I Built

C234: Consistent Hashing -- 7 implementations of consistent hashing algorithms:

1. **HashRing** -- Core consistent hash ring with virtual nodes (Karger et al.)
2. **WeightedHashRing** -- Nodes with different weights (proportional load)
3. **BoundedLoadHashRing** -- Google's bounded-load variant (2017), no node exceeds (1+epsilon) * avg
4. **JumpHash** -- Lamport/Thorup (2014), O(ln n) time, O(1) space, no ring
5. **RendezvousHash** -- Highest Random Weight / HRW hashing, O(n) per lookup
6. **MultiProbeHash** -- Appleton & O'Reilly (2015), 1 vnode per node, multiple probes
7. **MaglevHash** -- Google Maglev (2016), O(1) lookup table

Plus utility functions: `calculate_migration` and `measure_balance` for analysis.

## Key Design Decisions

- All algorithms share a common interface pattern: `add_node`, `remove_node`, `get_node`
- MD5 used as default hash (good distribution, deterministic, fast for non-crypto use)
- HashRing and Rendezvous support `get_nodes(key, count)` for replication
- BoundedLoad tracks load state -- requires explicit `release` or `reset_load`
- Maglev rebuilds lookup table on topology change (O(M*n) rebuild, O(1) lookup)
- JumpHash is static (append-only bucket model) -- different API reflects this

## Observations

- JumpHash has the best uniformity (CV < 0.10) but only supports append-only buckets
- Rendezvous has the best disruption properties -- removing node C only moves keys that were ON C
- MultiProbe achieves reasonable balance with far less memory than virtual nodes
- Maglev's O(1) lookup is ideal for high-throughput load balancers
- BoundedLoad prevents hotspots at the cost of tracking state
