- `class Name { init(x) { this.x = x; } method() { ... } }`

- `instanceof` builtin checks full inheritance chain

### Implementation

**3 new opcodes:** MAKE_CLASS, LOOKUP_METHOD, SUPER_INVOKE

**2 new tokens:** CLASS, SUPER

**3 new AST nodes:** ClassDecl, SuperCall, ExportClassDecl

**2 new runtime objects:** ClassObject (name, methods, parent), BoundMethod (instance, method, klass)

**Key design decisions:**
1. Instances are Python dicts with `__class__` key -- reuses existing INDEX_GET/INDEX_SET
2. Methods get implicit `this` first param (arity + 1)
3. Method dispatch: INDEX_GET on instance checks `__class__` chain, returns BoundMethod
4. Constructor: synchronous _run_until_return for init, pushes instance (not init return value)
5. Super: SUPER_INVOKE opcode looks up method in parent chain, injects `__super__` = found_class.parent

### Bugs Found and Fixed (implementation bugs, not design bugs)

1. **Stack corruption in constructor:** INDEX_SET in init body pushes values on shared stack. Fix: save/restore stack depth around _run_until_return.
2. **Super chain broken at 3+ levels:** CALL with raw FnObject doesn't inject `__super__`. Fix: added SUPER_INVOKE opcode for proper chain resolution.

### Test Bug Fixes (not implementation bugs)

- Closures use value-capture, not reference-capture (language semantics)
- `none` doesn't exist, use `false`
- Method names `from`/`print` collide with keywords -- renamed in tests
- Dict equality: instances are dicts, Python `==` compares content

## What I Learned

- Dict-based instances are elegant: reuse INDEX_GET/INDEX_SET, but need stack discipline in synchronous init
- Super resolution needs its own opcode -- can't piggyback on CALL because CALL doesn't know about class context
- BoundMethod pattern (instance + method + defining class) cleanly separates dispatch from invocation
- Implicit `this` as first param means no special calling convention needed
