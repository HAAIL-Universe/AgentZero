### C246: Transaction Manager (93 tests)

ACID transaction manager composing C240 (MVCC) + C241 (WAL) + C242 (Lock Manager).

- Unified transaction lifecycle: begin/commit/abort across all three subsystems
- Commit protocol: WAL commit -> MVCC commit -> Lock release
- Abort protocol: MVCC abort -> WAL abort -> Lock release
- MVCC snapshot reads (no locks for reads -- readers never block writers)
- Exclusive locks for writes via Lock Manager (prevents write-write conflicts)
- WAL logging before MVCC mutation (write-ahead protocol)
- Savepoints with partial rollback across all subsystems
- Batch operations (batch_put, batch_get, batch_delete)
- Context managers: Transaction (read-write) and ReadOnlyTransaction
- Crash recovery via ARIES
- Periodic checkpointing
- Garbage collection integration
- Configurable isolation levels, lock timeout, checkpoint interval

**Key design decision:** Reads use MVCC snapshots without locking. This is the fundamental
benefit of MVCC -- readers never block writers and writers never block readers. Only writes
acquire exclusive locks via the Lock Manager for write-write conflict prevention.

## Insight

The temporal integration closes a loop: scenario engine detects patterns, Shadow contextualizes
them against the user's history, and Speaker translates them into natural language the user
actually hears. Before this, temporal data only influenced simulation probabilities silently.
Now it surfaces as explicit behavioral observations -- which is what makes the system feel
genuinely aware rather than generically helpful.
