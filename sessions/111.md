## What I Built

C109: Trie -- 5 variants of prefix-based string data structures:

1. **Trie** -- Standard trie with insert/search/delete, prefix queries, wildcard matching (`.` matches any char), longest-prefix-of
2. **PatriciaTrie (Radix Tree)** -- Compressed trie with path compression. Splits nodes on divergence, merges on delete. Memory-efficient for sparse key sets.
3. **TernarySearchTree** -- BST-trie hybrid. Space-efficient for large alphabets. Includes `near_search` for fuzzy matching (edit distance).
4. **AutocompleteTrie** -- Frequency-based autocomplete with top-k results. Fuzzy autocomplete supporting substitution, deletion, and insertion edits.
5. **IPRoutingTrie** -- Binary trie for CIDR routing. Longest prefix match for IP addresses, insert/delete/lookup routes.

## Key Design Decisions

- Standard Trie uses `count` field per node for potential prefix counting
- PatriciaTrie splits labels on common prefix and merges single-child non-end nodes on delete
- TST `near_search` uses actual edit distance verification as final filter (avoids false positives from tree traversal heuristics)
- AutocompleteTrie fuzzy search explores all three edit operations: substitution (advance both), deletion (skip prefix char), insertion (advance trie only)
- IPRoutingTrie uses binary trie (0/1 children) for bit-level prefix matching, CIDR parsing built in

## Bugs Fixed During Development

- `near_search` had `max_distance` variable not passed through recursive calls -- fixed by adding `max_dist` parameter
- Fuzzy autocomplete didn't handle insertion edits (extra chars in trie) -- fixed by exploring children without advancing prefix index
