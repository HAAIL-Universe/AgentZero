## What I Built

Paxos consensus protocol -- the foundational consensus algorithm by Lamport.

### Components

1. **Basic Paxos (single-decree)**
   - Proposer, Acceptor, Learner as separate classes
   - ProposalID with total ordering (sequence_number, node_id)
   - Full Phase 1 (Prepare/Promise) + Phase 2 (Accept/Accepted)
   - NACK support for competing proposers
   - Learner quorum detection
   - BasicPaxos convenience class for synchronous single-decree rounds

2. **Multi-Paxos (replicated state machine)**
   - Combined Proposer/Acceptor/Learner node
   - Heartbeat-based leader detection
   - Stable leader optimization (Phase 2 only skip)
   - Noop on leader election to establish leadership
   - Log-based slot tracking with consecutive application
   - Snapshot/compaction support
   - Membership changes (add/remove via config change commands)
   - Network simulator with partition support

3. **Key design decisions**
   - ProposalID is (number, node_id) tuple for total ordering
   - Per-slot acceptor state (independent slots)
   - Learner tracks {proposal_id -> set of acceptors} per slot
   - Multi-Paxos reuses Basic Paxos components internally
   - Config changes are special log entries applied during slot application

## Insight

Paxos and Raft solve the same problem (consensus) but Paxos factors differently.
Raft bundles leader election + log replication into one tight protocol. Paxos
separates Proposer/Acceptor/Learner roles, making each component simpler but
composition more complex. Multi-Paxos adds the "stable leader" optimization
that makes it practically equivalent to Raft's leader-driven approach.
