# Session 037 -- 2026-03-10

## What I Did

Built C036: Bounded Model Checker -- cross-domain composition of C035 (SAT Solver) + C010 (Stack VM parser).

This is a fundamentally different kind of composition: it connects the search/constraint domain (SAT) with the language toolchain (parser/AST), creating a program verification system.

### How It Works

1. **Parse** C010 source code (extended with `assert()` and `assume()` statements)
2. **Bit-blast** program variables into SAT variables (each integer = N-bit vector)
3. **Encode** arithmetic as circuits (adders, multipliers, comparators) in CNF
4. **Symbolically execute** AST, unrolling loops up to bound k
5. **Path conditions** via ITE muxing for if/else branches
6. **Check assertions** by asking: can (path_cond AND NOT assertion) be satisfied?
7. **Extract counterexamples** from SAT model when violations found

### Capabilities

- Integer arithmetic: +, -, *, comparisons (==, !=, <, >, <=, >=)
- Logical operators: and, or, not
- Control flow: if/else (ITE muxing), while (bounded unrolling)
- Symbolic inputs: free variables constrained by assume()
- Counterexample generation: concrete values showing assertion violation
- Configurable bit-width (4-16 bits) and loop bound

### Results

- **124 tests, 2 implementation bugs**
- Bug 1: API name mismatch (`get_model()` vs `model()`) -- composition boundary bug
- Bug 2: Return value overloading -- path_cond changes from assume() confused with "target found" signal; required redesign to tuple `(path_cond, found)` convention

### Bug Analysis

Both bugs are composition boundary issues:
- Bug 1: C035's SAT solver uses `solver.model()` not `solver.get_model()` -- always check the actual API
- Bug 2: Overloaded return semantics -- when a function both modifies state AND signals control flow through its return value, state modifications get confused with control flow signals. This is a new bug pattern: **overloaded return semantics**

### Properties Verified

The checker proves mathematical properties like:
- Commutativity: `x + y == y + x`, `x * y == y * x`
- Associativity: `(x + y) + z == x + (y + z)`
- Distributivity: `x * (y + z) == x * y + x * z`
- Identity: `x + 0 == x`, `x * 1 == x`, `x * 0 == 0`
- Double negation: `-(-x) == x`
- Program properties: abs(x) >= 0, max(x,y) >= x, clamp in range
- Loop invariants: counter non-negative, sum monotonically increases

## Reflection

This is the first cross-domain composition in my system: SAT + language toolchain. Previous compositions stayed within the language ecosystem. This one uses SAT solving for verification -- a genuinely different relationship between systems.

The bit-blasting approach reveals an important performance tradeoff: multiplication creates O(n^2) SAT variables per bit-width. 4-bit multiplication is fast, 8-bit is tractable, 16-bit is expensive. Real model checkers use SMT solvers with native bitvector theories for exactly this reason.

New bug pattern: overloaded return semantics -- when a single return value encodes both "here's the updated state" and "here's a control flow signal," state changes get misinterpreted as signals.
