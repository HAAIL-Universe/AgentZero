### 2. DOMPurify Added to React Frontend

- This is defense-in-depth -- renderMarkdown already escapes `<`, `>`, `&` before processing, but DOMPurify catches edge cases

### 3. Security Headers Middleware (agent_zero_server.py)
Added `@app.middleware("http")` security_headers:
- `X-Content-Type-Options: nosniff` -- prevents MIME sniffing
- `X-Frame-Options: DENY` -- prevents clickjacking
- `Referrer-Policy: strict-origin-when-cross-origin` -- controls referrer leakage
- `Content-Security-Policy` on index page only:
  - `default-src 'self'` -- no external resources by default
  - `script-src 'self'` -- scripts only from same origin (React bundle)
  - `style-src 'self' 'unsafe-inline'` -- needed for React inline styles
  - `connect-src 'self' ws: wss:` -- WebSocket connections
  - `object-src 'none'` -- no plugins
  - CSP NOT applied to API endpoints (would break CORS)

### 4. CORS Tightened
- Changed `allow_headers=["*"]` to `allow_headers=["Authorization", "Content-Type"]`
- Only headers actually needed by the frontend

### 5. Legacy Template XSS Fixes (agent_zero.html)

- Added `esc()` to all onclick IDs: req.id, g.id, c.id (prevents quote breakout)

## Key Insight

The research paper was written against the legacy `agent_zero.html` (4101 lines), but the active frontend is the React SPA in `agent_zero-ui/`. The real XSS vector in React was `dangerouslySetInnerHTML` with `renderMarkdown()` output (2 instances in AppShell.tsx). DOMPurify is the correct fix -- it sanitizes at the boundary between markdown processing and DOM injection.
