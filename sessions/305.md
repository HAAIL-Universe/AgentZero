## What I Did

Extracted 4 custom hooks from the monolithic App.tsx (1188 lines -> 677 lines, 43% reduction):

### useAuth (142 lines)
- Auth state: token, user, mode, email, password, display name, error, remember device
- Token persistence to localStorage/sessionStorage based on remember-device preference
- Login/register/logout flows
- Auto-fetch current user on token change with expired-token handling

### useVoice (217 lines)
- Recording state: MediaRecorder lifecycle, blob capture, WS audio send
- Speech synthesis: SpeechSynthesisUtterance with wave animation
- Wave bar animation with pulse effect
- Voice overlay and status management

### useDesktopRuntime (208 lines)
- Desktop runtime config CRUD (save, connect, disconnect, refresh, publish)
- Auto-load status on mount and when settings panel opens
- Draft/dirty tracking for unsaved changes
- Callback pattern for connect/disconnect to allow socket epoch bumping

### useOnboarding (208 lines)
- Onboarding state, answers, step navigation
- Orientation banner dismiss/restore
- Voice transcription for onboarding answers
- Persistence to backend with error handling

## Design Decisions

1. **Callback pattern for cross-hook communication** -- logout calls `auth.logout(cleanupFn)` where cleanupFn stops voice/recording. Desktop connect/disconnect take callbacks for socket epoch bumping. This avoids circular dependencies between hooks.

2. **Voice hook exposes Dispatch<SetStateAction<string>>** for `setLatestAssistantVoice` -- needed because the WS `token` handler uses functional updates (`prev => prev + token`).

3. **WebSocket + data fetching stay in App.tsx** -- the socket message handler touches too many state domains (messages, reasoning, voice, generating) to cleanly extract without an event bus or context. Data fetching is simple enough that extraction would add indirection without reducing complexity. These are candidates for Phase 2 if needed.
