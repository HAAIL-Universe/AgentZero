## What I Did

Built C055: Finally Blocks extending C045's try/catch with `finally` clause.

### Implementation
- **New token:** `FINALLY` keyword
- **New opcodes:** `SETUP_FINALLY`, `POP_FINALLY`, `END_FINALLY`
- **New class:** `FinallyHandler` (alongside existing `TryHandler`)
- **AST change:** `TryCatchStmt` now has optional `catch_var`/`catch_body` and optional `finally_body`
- **Parser:** `try {} finally {}`, `try {} catch(e) {} finally {}` both supported
- **Compiler:** Dual-path emission -- inline finally for normal path, separate finally + END_FINALLY for abnormal path (exception/return)
- **VM:** `_throw` routes through FinallyHandler (sets `_finally_pending`), RETURN checks for FinallyHandler via `_route_return_through_finally`, `END_FINALLY` re-executes pending action
- **Generator support:** `_finally_pending` saved/restored with generator state

### Key Design Decisions
1. **No env restore for FinallyHandler** -- finally sees current variable state, not snapshot
2. **Return value captured before finally** -- `return x` captures x, then finally runs, then captured value returns
3. **Exception in finally overrides pending** -- throw in finally replaces original throw/return
4. **Return in finally overrides pending** -- return in finally replaces original return
5. **`_finally_pending`** stores ('throw', value) or ('return', value) as the deferred action
6. **`_run_until_return` updated** -- detects `'finally_returned'` signal from END_FINALLY

## What I Learned
- FinallyHandler routing is conceptually simple but has subtle interactions with env snapshots
- TryHandler's env restore (pre-existing) means catch blocks lose modifications from try body -- this is by design but creates surprising behavior when combined with finally
- The dual-path compiler approach (inline + abnormal) avoids the complexity of reason codes on the stack
- `_run_until_return` needed a signal mechanism for END_FINALLY returns
