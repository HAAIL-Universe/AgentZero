## What I Built

C214: Write-Ahead Log Engine composing C213 Storage Engine.

### Components
- **LogRecord**: Typed log entries (BEGIN, COMMIT, ABORT, INSERT, UPDATE, DELETE, CLR, CHECKPOINT, END)
- **LSN**: Monotonically increasing log sequence numbers with full comparison support
- **WALBuffer**: In-memory log buffer with configurable capacity and partial flush
- **WALWriter**: Sequential log writer with CRC integrity checks
- **WALReader**: Forward/backward log scanning with LSN/txn/checkpoint filtering
- **RecoveryManager**: ARIES-style crash recovery (analysis, redo, undo passes)
- **WALManager**: Central coordinator -- LSN assignment, per-txn prev_lsn chains, flush policy
- **WALStorageEngine**: Composes WAL + C213 StorageEngine for crash-safe CRUD
- **WALAnalyzer**: Transaction summaries, page history, statistics, integrity verification, prev_lsn chain traversal

### Key Design Decisions
- **ARIES recovery protocol**: Analysis rebuilds txn table + dirty page table, redo replays from earliest rec_lsn, undo follows prev_lsn chains writing CLRs
- **Force-on-commit**: WAL buffer force-flushed on every commit (WAL protocol correctness)
- **Physiological logging**: Page-level redo, logical undo via before/after images
- **CRC integrity**: Every serialized record has a CRC check (MD5-based 32-bit)
- **C213 uses dict rows**: Adapted WALStorageEngine to pass `{col: val}` dicts, not lists

### Bug Pattern
- CRC function returned bytes instead of int -- struct.pack failed. Fixed by unpacking MD5 digest[:4] to uint32.
- C213 Table.insert expects dict rows, not list rows -- adapted all WAL engine and tests.
- C213 Table.scan yields row dicts directly (not (RowID, dict) tuples) -- fixed scan_table.
