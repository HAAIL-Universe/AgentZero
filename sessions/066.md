## Design

For-await is lazy async iteration -- values consumed one at a time from async generators, unlike the eager `for-in` which collects everything into an array first.

### Syntax
- `for await (x in asyncGen()) { body }` -- basic
- `for await ([a, b] in asyncGen()) { body }` -- array destructuring
- `for await ({name, age} in asyncGen()) { body }` -- hash destructuring
- `for await ([first, ...rest] in asyncGen()) { body }` -- rest patterns
- Supports break, continue, early return, nested for-await

### Implementation changes:
1. **Opcode**: `ASYNC_ITER_NEXT addr` -- calls next() on async generator, awaits the promise (with drain), if done jumps to addr, else pushes value
2. **AST**: `ForAwaitStmt` and `ForAwaitDestructure` nodes
3. **Parser**: Detects `for await (` after FOR token -- consumes AWAIT, then parses normally, returns ForAwait* nodes instead of ForIn* nodes
4. **Compiler**: `compile_ForAwaitStmt` and `compile_ForAwaitDestructure` -- evaluate iterable, store agen, loop with ASYNC_ITER_NEXT opcode
5. **VM**: ASYNC_ITER_NEXT handler -- calls _next_async_generator, drains async queue, checks done, throws rejection through handler stack

### Key decisions:
- One new opcode (ASYNC_ITER_NEXT) instead of pure compiler desugaring, because detecting "done" requires Python-level access to `agen.done`
- Error propagation uses `_throw()` to route through handler stack properly (not wrapping in VMError string)
- try-catch env restore means in-loop variable mutations are lost on error (consistent with existing behavior)
- All existing for-in behavior unchanged -- the parser only activates for-await when AWAIT token follows FOR
