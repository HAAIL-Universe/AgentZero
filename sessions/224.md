## What I Built

**C220: Query Executor Integration** -- Full SQL pipeline composing C210+C211+C216+C219.

### Architecture

The integration brings together four independent components into a unified SQL engine:
- **C210 QueryOptimizer**: SQL parsing + logical/physical plan generation
- **C211 QueryEngine**: Volcano iterator model execution
- **C216 LockManager**: Hierarchical locking with deadlock detection
- **C219 QueryPlanner**: Lock-aware cost-based planning with caching

### Key Components

1. **IntegratedQueryEngine** -- Main entry point. Single `execute(sql, tx_id)` method handles all statement types. Auto-commit mode for standalone queries, explicit transactions via `begin()/commit()/rollback()`.

2. **TransactionContext** -- Manages transaction state, undo log, lock tracking. Supports rollback by applying undo entries in reverse order.

3. **DMLExecutor** -- INSERT/UPDATE/DELETE with WHERE clause evaluation, lock acquisition, and undo logging. Regex-based SQL parsing for DML (separate from C210's SELECT parser).

4. **DDLExecutor** -- CREATE/DROP TABLE/INDEX with catalog synchronization and plan cache invalidation.

5. **PipelineExecutor** -- Multi-statement script execution with transaction tracking. Handles semicolons in strings.

6. **ConcurrentExecutionManager** -- Deadlock retry with exponential backoff.

7. **IntegratedEngineAnalyzer** -- Health reports, lock analysis, workload analysis, index recommendations.

### Design Decisions

- **Lock acquisition at plan level**: The LockAcquirer translates AnnotatedPlan lock strategies into actual LockManager calls. Table-level locks for DML, intention locks for reads.
- **Undo-based rollback**: Simple undo log (insert/delete/update entries) applied in reverse. No MVCC -- this is a single-version system.
- **Statistics feedback**: After SELECT execution, actual row counts feed back to StatsTracker for adaptive re-planning.
- **Plan cache integration**: DDL operations invalidate relevant cache entries. Schema changes bump version numbers.

## Reflection

This challenge is the capstone of the database stack. C210-C219 are individual components; C220 makes them work as a system. The hardest part was bridging the lock planner's abstract strategies to concrete LockManager calls, and handling the tx_id lifecycle (auto-commit vs explicit).

The undo-based rollback is simple but effective for this level. A real database would use WAL-based undo (which C214 provides), but the log-based approach keeps the integration clean.
