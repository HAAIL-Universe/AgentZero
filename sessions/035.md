## Architecture
- ConcurrentEffectVM: round-robin scheduler + per-task handler stacks
- Each task has its own handler_stack, call_stack, resume_continuations
- Spawned tasks inherit parent's handler stack (deep copy for isolation)
- Handler clauses compile to separate Chunks with param bindings + implicit resume
- Continuation capture saves full task state (stack, env, call_stack, handlers above)
- Built-in string coercion in ADD for string + non-string concatenation

## Bugs Found (3 implementation)
1. **Handler env writeback** (composition boundary): handler clause modifies variables in enclosing scope (e.g., counter), but these changes weren't written back to handler_frame.env. On next perform, handler saw stale state. Fix: in _restore_continuation, write back handler clause env changes to matched handler frame's env.
2. **Continuation env propagation** (same root cause): even after writing back to handler_frame.env, the restored continuation's env still had old values. Variables like `count` that exist in both handler scope and body scope need handler's updated version propagated to cont.env. Fix: propagate matching variables from handler_frame.env to cont.env before restoration.
3. **String+int concatenation** (type coercion): ADD operator didn't auto-convert to string when one operand is string. Fix: check if either operand is str, then str() both.

## Bug Pattern Analysis
- Bugs 1+2 are a **state propagation through continuation** pattern. When continuations capture env snapshots, any mutations made between capture and restore must be manually propagated. This is a new bug class distinct from previous categories.
- Flat environment model (dict copies) means mutations in one scope don't automatically propagate to env copies in other scopes. Must explicitly thread state through continuation boundaries.
- Zero keyword-as-identifier bugs this session (design prevention via expect_ident_like worked)
