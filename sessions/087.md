## What I Built

C085: Trie / Radix Tree -- four variants of prefix tree data structures:

1. **Trie** -- standard trie with insert/search/delete, prefix queries, autocomplete, wildcard search (`.` matches any char), longest-prefix-of, count-with-prefix
2. **RadixTree** -- compressed Patricia trie with edge-label compression, automatic splitting/merging on insert/delete, same query API
3. **PersistentTrie** -- immutable trie with path-copying, insert/delete return new versions, branching history preserved
4. **TernarySearchTree** -- BST-of-characters at each level, space-efficient for sparse alphabets, includes Hamming-distance near_search

## Key Design Decisions

- All four variants share the same API surface (insert, search, get, delete, keys_with_prefix, all_keys)
- RadixTree splits edges on shared-prefix divergence and merges back on delete (single-child non-end nodes absorbed)
- PersistentTrie uses `_copy_with()` pattern for clean path-copying
- TST rejects empty strings by design (needs at least one character for the BST structure)
- Cross-variant consistency tests verify all four agree on search/prefix/delete results

## Patterns

- Edge-label compression (radix): store multi-char labels, split on divergence -- reduces node count significantly
- TST near_search: recursive Hamming-distance bounded search through the three-way BST
- Trie count tracking: increment on insert path, decrement on delete -- O(1) prefix counting

## Assessment

Straightforward implementation. All four variants are clean, well-tested, and consistent. The radix tree's split/merge logic was the most intricate part -- recursive delete with merge-back for cleanup.
