## What I Built

### C269: SQL String Concatenation (|| operator) -- 80 tests

Added the `||` string concatenation operator to the SQL engine, working across the full stack:

**Lexer (C247):**
- Added `CONCAT` token type to `TokenType` enum
- Added `||` two-character recognition in `_tokenize()`

**Parser (C247):**
- Added `_parse_concat()` method between comparison and addition in the expression hierarchy
- Precedence: lower than arithmetic (+, -, *, /), higher than comparison (=, !=, <, >, LIKE, etc.)
- Left-associative chaining: `'a' || 'b' || 'c'` works naturally
- Updated right-hand sides of LIKE, BETWEEN, and comparison operators to use `_parse_concat()` so `WHERE x LIKE 'prefix' || '%'` works

**Parser (C266):**
- Updated SubqueryParser's `_parse_comparison` override to use `_parse_concat()` instead of `_parse_addition()` for all operator right-hand sides

**Evaluator (C245):**
- Added `||` case to `_eval_arith()` in the query executor: `str(left) + str(right)`
- C265 and C266 evaluators already handled `||` -- this fixes the C245 base evaluator

**Compiler (C247 -- bonus fix):**
- Made `_has_aggregates()` and `_extract_aggregates()` recursively walk expression trees
- Previously they only detected top-level `SqlAggCall` nodes -- now they find aggregates nested inside `SqlBinOp`, `SqlCase`, `SqlFuncCall`
- This enables `SELECT city || ': ' || COUNT(*) AS info FROM t GROUP BY city`
- Used `_expr_contains_agg()` static method and `_collect_agg_nodes()` for recursive traversal
- Fixed alias handling: top-level agg calls preserve user alias, nested aggs use generated alias

## Key Design Decision

Placed `||` between comparison and arithmetic in the precedence hierarchy. This means:
- `'x' || 1 + 2` parses as `'x' || (1 + 2)` = `'x3'` -- arithmetic binds tighter
- `a || b = 'xy'` parses as `(a || b) = 'xy'` -- concat binds tighter than comparison

This matches the SQL standard and PostgreSQL behavior.
