## What I Built

**C226: Database Replication** -- Primary-replica replication system composing C223 (Connection Pool) + C201 (Raft Consensus).

### Components (10):
1. **WALEntry** -- Checksummed write-ahead log entries with tamper detection
2. **WALShipper** -- Ships WAL entries with replication slots, pending tracking, compaction
3. **DataStore** -- In-memory data store with INSERT/UPDATE/DELETE/DDL operations
4. **ReplicationNode** -- Node with role (primary/replica), promote/demote lifecycle
5. **ReplicationCluster** -- Full cluster with sync/async/semi-sync modes, failover, snapshot catch-up
6. **RaftReplicationCluster** -- Raft-based cluster using C201 for automatic leader election
7. **ConflictDetector** -- Write-write conflict detection with last-writer-wins and priority resolution
8. **LagMonitor** -- Replication lag tracking with warn/critical alerts
9. **StreamingReplicationManager** -- Continuous streaming with catch-up, batching, error recovery
10. **ReadWriteSplitter** -- Routes reads to replicas, writes to primary, round-robin distribution

### Key Design Decisions:
- WAL entries carry MD5 checksums for integrity verification
- Three replication modes: sync (all confirm), semi-sync (N confirm), async (fire-and-forget)
- Replication slots track per-replica progress with sent_lsn and confirmed_lsn
- Failover promotes replica with highest applied_lsn, re-registers other replicas
- Raft integration uses submit_and_commit for consensus-based writes
- Streaming manager supports batch-based catch-up with configurable batch size

### Bugs Fixed During Development:
- RaftCluster takes node_ids iterable, not integer count
- Raft role is an Enum (Role.LEADER), not string -- fixed with .value comparison
- Replication slot start_lsn must begin at replica's position, not current WAL position
- Used submit_and_commit instead of manual log.append for proper Raft replication
