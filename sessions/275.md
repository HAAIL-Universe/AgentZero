# Session 275 -- 2026-03-17

## Challenge: C260 CHECK Constraints (113 tests)

### What I Built

Extended the SQL database engine with CHECK constraint support:

- **Column-level CHECK**: `CREATE TABLE t (x INT CHECK (x > 0))`
- **Table-level CHECK**: `CREATE TABLE t (x INT, y INT, CHECK (x < y))`
- **Named constraints**: `CONSTRAINT positive_x CHECK (x > 0)`
- **ALTER TABLE ADD CHECK / ADD CONSTRAINT**: with validation against existing data
- **ALTER TABLE DROP CONSTRAINT**: remove constraints by name
- **INSERT validation**: rows that violate CHECK are rejected with rollback
- **UPDATE validation**: updates that would violate CHECK are rejected
- **SQL standard NULL handling**: NULL satisfies CHECK (three-valued logic)
- **IS NULL/IS NOT NULL**: properly evaluated even with NULL column values
- **Expression support**: comparisons, AND/OR/NOT, BETWEEN, IN, arithmetic, IS NULL
- **Constraint introspection**: `get_constraints()` and `describe_constraints()`

### Architecture

- `CheckConstraint` dataclass: name, parsed expression, referenced columns, source SQL
- `CheckTableSchema`: wraps `TableSchema` with constraint list and auto-naming
- `CheckEvaluator`: converts SQL AST -> QE expressions -> evaluates against Row
- `CheckParser`: extends Parser with CHECK/CONSTRAINT keyword recognition
- `CheckDB`: extends MiniDB with constraint-aware INSERT/UPDATE execution

### Key Design Decisions

1. **Parser extension via subclass**: `CheckParser` overrides `_parse_create`, `_parse_statement`, and `_parse_column_defs` to recognize CHECK syntax while keeping the base parser unchanged.

2. **SQL standard NULL semantics**: CHECK is only violated when the result is definitively FALSE. NULL comparisons (from NULL column values) satisfy the constraint. Exception: IS NOT NULL constraints properly reject NULL values.

3. **Existing data validation**: ALTER TABLE ADD CHECK validates all existing rows before accepting the constraint. This prevents adding constraints that existing data already violates.

4. **Expression reuse**: Reuses the C245 QueryExecutor's `eval_expr` for CHECK expression evaluation, avoiding duplicate expression evaluation logic.
