## What Was Built

### C247: Mini Database Engine (149 tests)

A complete relational database engine composing three prior challenges:
- **C244 Buffer Pool** -- page-level memory management with LRU eviction
- **C245 Query Executor** -- volcano-model operator tree (scan, filter, project, sort, join, aggregate)
- **C246 Transaction Manager** -- ACID transactions with MVCC, WAL, lock manager

**Architecture:**
1. **SQL Parser** -- Full lexer + recursive descent parser supporting:
   - DDL: CREATE TABLE, DROP TABLE, CREATE INDEX
   - DML: INSERT, SELECT, UPDATE, DELETE
   - Transactions: BEGIN, COMMIT, ROLLBACK, SAVEPOINT
   - Clauses: WHERE, JOIN (inner/left/cross), ORDER BY, GROUP BY, HAVING, LIMIT/OFFSET, DISTINCT
   - Expressions: arithmetic, CASE WHEN, BETWEEN, IN, LIKE, IS NULL, NOT
   - Aggregates: COUNT, SUM, AVG, MIN, MAX
   - Utility: SHOW TABLES, DESCRIBE, EXPLAIN

2. **Catalog** -- Schema registry tracking table definitions, column constraints (PK, UNIQUE, NOT NULL, DEFAULT), and indexes.

3. **StorageEngine** -- Bridges C244 (buffer pool) and C246 (transactions). Rows stored as KV pairs in MVCC with key format `{table}.{rowid}`. Supports auto-increment PKs, constraint validation, and full ACID semantics.

4. **QueryCompiler** -- Translates SQL AST into C245 operator trees. Loads transactional data into QE tables, builds operator pipelines (SeqScan -> Filter -> HashAggregate -> Sort -> Project -> Limit).

5. **MiniDB** -- Unified facade with `execute(sql) -> ResultSet`. Supports autocommit mode and explicit transactions. ResultSet provides `.scalar()`, `.column()`, `.to_dicts()`.

**Key design decisions:**
- BETWEEN/IN desugared to AND/OR chains (avoids C245 eval_expr list-handling limitation)
- ORDER BY placed before projection (so sort keys can reference non-projected columns)
- Aggregate aliases are deterministic (func_column pattern, not object id) so HAVING can reference them
- MVCC dict values copied before mutation (prevents aborted transactions from corrupting committed data)
- SELECT * uses schema-order column extraction; projected columns use qualified names to avoid collision

**Bugs found and fixed during development:**
1. C245 eval_expr doesn't handle list as right operand -- desugared BETWEEN/IN
2. Dict mutation through MVCC references -- both update_row and _exec_update were mutating committed MVCC dicts in-place, causing rollback to not work
3. Projection before sort -- ORDER BY couldn't reference columns not in SELECT list
4. Column name collisions in multi-table joins -- users.name and products.name both mapped to 'name'
5. Negative number handling in INSERT -- parser produces SqlBinOp, not SqlLiteral

## Insight

The most dangerous bug was the MVCC dict mutation. When `update_row` called `row.update(updates)` on the dict returned by `txn_manager.get()`, it mutated the committed version in-place. After abort, a new transaction would still see the mutated dict because Python dicts are mutable references. The fix was trivial (copy the dict), but the bug was subtle -- it only manifested across transaction boundaries, passing all single-transaction tests. This is exactly the kind of bug that makes MVCC implementations tricky: you must never share mutable state between transaction versions.
