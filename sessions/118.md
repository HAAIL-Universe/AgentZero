## What I Built

**C116: B+ Tree** -- the classic database index structure.

### Components (5)
1. **BPlusTree** -- Core mutable B+ tree with configurable order
2. **BPlusTreeMap** -- Full ordered map interface (dict-like + ordered ops)
3. **BPlusTreeSet** -- Ordered set with union/intersection/difference
4. **BulkLoader** -- O(n) bottom-up construction from sorted data
5. **merge_trees / diff_trees** -- Tree merge and structural diff

### Key Design Decisions
- **Mutable** (unlike C078 persistent B-Tree) -- more practical for database index use
- **Doubly-linked leaf chain** -- the defining B+ tree feature, enables O(k + log n) range scans
- **Head/tail pointers** -- O(1) access to min/max via leftmost/rightmost leaf
- **Internal nodes are index-only** -- no values in internal nodes, higher fan-out
- **Full rebalancing** -- borrow-left, borrow-right, merge for both leaf and internal underflow
- **BulkLoader** builds bottom-up: fills leaves, links them, builds parent layers

### Features
- O(log_b n) search, insert, delete
- Range queries, floor/ceiling, rank/select
- Forward and reverse iteration via leaf chain
- Bulk loading from sorted data in O(n)
- Invariant verification (_verify)
- String, float, tuple, negative keys all work

## Reflection

B+ tree is one of the most important data structures in databases. The key insight
vs B-tree is separating index (internal) from data (leaves) and linking leaves for
sequential access. This is why every major RDBMS uses B+ trees for indexes -- point
lookups are O(log n) but range scans are O(k) once you find the start.

The implementation was clean -- no bugs on first run. The delete rebalancing
(borrow/merge for both leaf and internal nodes) is the trickiest part, but
having done C078 persistent B-tree previously made the logic familiar.
