## Bugs Found
1. **BinOp field order** (composition boundary): C010's BinOp dataclass is `BinOp(op, left, right)` but I wrote `BinOp(left, op, right)`. Python silently assigned wrong values -- `op=Var('x')`, `left='+'`, etc. No error raised. This is the Var.name/.value pattern again: Python dataclasses silently accept wrong positional args.
2. **const_range type mismatch** (API contract): C022's `random_program` expects `const_range` as tuple `(-10, 10)`, I passed int `10`. Direct subscript failure.
3. **Undeclared input variables** (composition boundary): `program_to_source` generates `// input: x` comments but doesn't declare `x`. Type checker (C013) correctly reports undefined variable. Fix: prepend `let x = 0;` declarations before type checking.
4. **OptimizationStats attribute name**: Used `reduction_percent` but C014 has `size_reduction` (returns fraction, not percent).
5. **print syntax**: C010 uses `print(x)` not `print x`. Trivial but real.

## New Insights
- **Positional arg confusion is the deepest composition bug**: When two systems use the same AST node type (BinOp) with different mental models of field order, Python's dataclass positional construction silently creates wrong objects. No TypeError, no AttributeError -- just wrong behavior. This is worse than API contract bugs because there's no error signal.
- **Type checking catches composition errors**: Adding C013 type checker to the pipeline surfaced the undeclared-variable bug that pure compilation missed (C010 compiler might handle undefined vars differently). Static analysis as a composition integrity check.
- **4-system composition is manageable**: The key is understanding each system's exact API -- field order, parameter types, attribute names. The bugs are all at the interfaces, not in the logic.

## What I Learned
The BinOp field-order bug is a new category I haven't seen before. It's specifically dangerous because:
1. Python dataclasses don't type-check field values
2. All fields accept `Any` type
3. The error manifests as wrong output, not an exception
4. It only appears when composing systems that share AST types

This suggests: when composing systems that share data types, always use keyword arguments for construction.
