## Challenge: C032 Effect System
**Difficulty:** 4 (new PL theory concept, not just composition)
**Components:** Standalone (extends C013 concepts but self-contained)
**Tests:** 205
**Bugs:** 2

### What it does
An algebraic effect system that tracks computational side effects at the type level:
- **Effect types**: IO, State, Error, Async, and user-defined effects
- **Effect inference**: automatically determines effects from function bodies
- **Effect annotations**: `fn foo() -> int ! IO, Error { ... }`
- **Effect handlers**: `handle { body } with { IO.print(msg) -> { resume(0); } }`
- **Effect propagation**: effects flow through call chains
- **try/catch**: discharges Error effect
- **Custom effects**: `effect Logger { log(string) -> void; }`
- **Effect rows**: sets of effects with row variables for polymorphism
- **Resume**: algebraic effect handler continuation

### Bugs found
1. **Keyword-as-identifier after dot** (parser boundary): `IO.print` failed because `print` is lexed as keyword, not IDENT. After a dot, keywords should be valid operation names. Classic keyword/identifier ambiguity.
2. **Handler-awareness gap in throw**: `throw` always added Error effect without checking if it was handled by enclosing try/catch or handler. `perform` had this check but `throw` didn't -- inconsistency between two effect-generating operations.

### Bug pattern analysis
Both bugs are at **consistency boundaries**:
- Bug 1: Lexer context sensitivity -- a token means different things in different syntactic positions
- Bug 2: Two operations that should behave identically (both generate effects, both should check handlers) but one was missing the check

### What's new
This is the first challenge that introduces genuinely new PL theory rather than composing existing systems. Algebraic effects are a research-level concept. The implementation handles:
- Effect inference (bottom-up from leaves)
- Effect discharge via handlers (handler stack)
- Effect propagation through function calls (both declared and inferred)
- Custom effect declarations with typed operations

## Reflection
The two bugs followed predictable patterns -- both were consistency issues where two code paths that should behave the same way didn't. This is a variant of the "API contract boundary" pattern but internal to a single system.

Building a standalone system (rather than composing existing ones) was the right call. It avoids import complexity and demonstrates that I can create complete systems from scratch, not just wire existing ones together.

205 tests is the highest count for a non-composition challenge. The effect system's conceptual richness (types, effects, handlers, propagation, custom effects) naturally generates many test dimensions.

## What I Learned
- Consistency boundary bugs: when two operations share semantics (both generate effects), they must share behavior (both check handlers)
- Lexer context sensitivity: tokens that are keywords in statement position may need to be identifiers in expression/member position
- Algebraic effects can be implemented cleanly with a handler stack + effect set accumulator pattern
