## What I Built

**C249: Stored Procedures & User-Defined Functions** -- Full stored routine support for the Mini Database Engine (C247).

### Features
- **CREATE FUNCTION**: Scalar UDFs with typed parameters, SQL body, RETURN
- **CREATE PROCEDURE**: Multi-statement procedures with IN/OUT/INOUT parameters
- **CALL**: Execute procedures and functions, return OUT/INOUT parameter values
- **DROP FUNCTION/PROCEDURE**: With IF EXISTS support
- **CREATE OR REPLACE**: Replace existing routines
- **SHOW FUNCTIONS/PROCEDURES**: List all stored routines
- **Control flow**: IF/ELSEIF/ELSE/END IF, WHILE/DO/END WHILE
- **Variables**: DECLARE with type and DEFAULT, SET assignment
- **SELECT INTO**: Query results into local variables
- **Loop control**: LEAVE (break) and ITERATE (continue) with labels
- **Recursion**: Recursive function calls with depth limiting (max 100)
- **Exception handling**: DECLARE CONTINUE/EXIT HANDLER FOR SQLEXCEPTION
- **Nested calls**: Functions calling functions, procedures calling procedures
- **UDFs in SELECT**: Constant UDF calls resolved in queries
- **17 built-in functions**: abs, upper, lower, length, concat, coalesce, ifnull, nullif, substr, replace, trim, round, floor, ceil, mod, power, sqrt, greatest, least

### Architecture

**Extended Lexer** (`ProcLexer`): Adds 20+ procedure keywords (FUNCTION, PROCEDURE, RETURNS, RETURN, CALL, DECLARE, WHILE, DO, etc.) as `ProcToken` objects alongside C247's `Token`/`TokenType` system.

**Extended Parser** (`ProcParser`): Inherits from C247 `Parser`, adds:
- `_parse_body_statement()` -- dispatches DECLARE, SET, IF, WHILE, RETURN, LEAVE, ITERATE, CALL, SELECT INTO
- `_parse_begin_end()` -- BEGIN...END block parsing
- `_parse_if()` with ELSEIF chains
- `_parse_while()` with optional labels
- Full expression parser with modulo (%) support

**Execution Engine** (`ProcExecutor`):
- `call_function()` / `call_procedure()` with parameter binding and type coercion
- `_exec_body()` with handler collection and error dispatch
- `_exec_if()` / `_exec_while()` with proper signal propagation (ReturnSignal, LeaveSignal, IterateSignal)
- `_substitute_vars_in_stmt()` for SQL DML execution with local variable substitution
- `_eval_sql_expr()` for expression evaluation against local variable scope

**Extended DB** (`ProcDB`): Inherits from C247 `MiniDB`, adds `RoutineCatalog` for function/procedure storage, UDF resolution in SELECT queries.

### Key Design Decisions
1. **ProcToken vs TokenType**: Procedure keywords use string-typed tokens to avoid modifying C247's enum. Cross-matching in expect/match handles both types.
2. **Variable substitution**: Rather than threading variable scope through the query pipeline, substitute variable refs with literals before compiling SQL. Reuses all C247 machinery unchanged.
3. **Signal-based control flow**: RETURN, LEAVE, ITERATE use exception-based signals. Clean, composable, handles nested blocks correctly.
4. **Aggregate lookahead**: COUNT/SUM/etc. only parsed as aggregates when followed by `(`. Otherwise treated as variable names (e.g., `count` as a loop variable).

### Bugs Encountered During Testing (all fixed before completion)
- `TokenType.DECLARE` doesn't exist -- DECLARE is a proc keyword string, not C247 keyword
- `END IF` fails -- IF is ProcToken `'IF'`, not `TokenType.IF`. Fixed with cross-matching in expect().
- `add` as function name -- `TokenType.ADD` not accepted as identifier. Fixed by expanding keyword-as-ident set.
- `DROP FUNCTION IF EXISTS` -- IF keyword not matched. Fixed with cross-type check.
- `CatalogError` not caught by handler -- not a subclass of `DatabaseError`. Fixed by catching multiple exception types.
- `count` as variable name -- tokenized as `TokenType.COUNT`, triggers aggregate parsing. Fixed with lookahead for `(`.
