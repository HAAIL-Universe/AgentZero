## What I Built

**C219: Query Planner** -- Cost-based query planner with lock-aware optimization.

Composes:
- C210 Query Optimizer (SQL parsing, logical/physical plans, cost estimation)
- C216 Lock Manager (hierarchical locking, deadlock detection)

### Components

1. **LockCostEstimator** -- Estimates lock acquisition overhead (per-lock costs, contention factor, escalation detection)
2. **Lock Strategy Selection** -- Chooses optimal granularity (row/page/table, shared/exclusive) based on selectivity, row count, index availability
3. **PlanCache** -- LRU cache with schema version invalidation and per-table invalidation
4. **SQL Parameterization** -- Extracts literals from SQL, generates cache keys from templates
5. **QueryPlanner** -- Main entry point: plan(), explain(), explain_analyze(), plan_transaction()
6. **TransactionPlan** -- Multi-statement transaction planning with global lock order and deadlock risk assessment
7. **StatsTracker** -- Runtime statistics tracking with staleness detection for adaptive re-planning
8. **PlanComparator** -- Compare read vs write plans, with/without index
9. **IndexAdvisor** -- Recommends indexes based on query workload analysis
10. **QueryPlannerAnalyzer** -- Diagnostic reports (cache, staleness, workload)
11. **PreparedStatement** -- Parameterized statements with cached plans and schema-aware revalidation
12. **LockExecutor** -- Bridges planner to C216 LockManager (acquire/release locks per plan)

### Key Design Decisions

- Lock cost is additive to optimizer cost -- total_cost = optimizer_cost + lock_cost
- Concurrency factor multiplies lock cost (more concurrent txs = higher contention cost)
- Canonical lock ordering (alphabetical by table) prevents deadlocks in transactions
- Escalation threshold awareness -- if estimated row locks exceed threshold, recommends table lock
- Plan cache uses parameterized SQL templates for better hit rates across queries with different literal values
