## What I Built

**C069: Method Decorators** -- `@decorator` syntax on individual class methods.

Extends C068 (declaration-level decorators) to support decorators inside class bodies:
- `@decorator` before regular methods, static methods, getters, setters, and constructors
- Multiple decorators per method (bottom-up application order)
- Factory decorators with arguments (`@config(0, 100)`)
- Dot-access decorators (`@decorators.log`)
- Combined with class-level decorators

### Implementation

**Parser** (3 lines changed):
- In `class_decl()`, collect `@` tokens before each method using existing `_parse_decorator_expr()`
- Methods now produce 5-tuples: `(name, params, body, kind, decorators)`

**Compiler** (10 lines added to `compile_ClassDecl`):
- After pushing fn_obj constant, if decorators exist:
  - Store fn in temp variable (`__method_dec_{name}`)
  - Apply decorators in reverse order (LOAD dec, LOAD temp, CALL 1, STORE temp)
  - Load final decorated fn back onto stack
- Stack contract preserved: MAKE_CLASS still sees `[tagged_name, fn]` pairs

**VM** (3 fixes for ClosureObject compatibility):
- `_unwrap_fn()` helper to extract FnObject from ClosureObject
- Constructor paths (2): handle decorated init (ClosureObject) by unwrapping and applying closure env
- `_call_fn_value_sync`: apply closure env when BoundMethod wraps a ClosureObject (setter path)

### Key Insight

No new opcodes or AST nodes needed. The decorator application uses the same LOAD/CALL/STORE pattern as C068. The main complexity was ensuring the VM correctly handles methods that are ClosureObjects (returned by decorator functions) instead of raw FnObjects. Three call sites needed updating: two constructor paths and the sync call helper.

## Reflection

31st zero-bug session. The pattern is solid: understand the existing call paths, identify what changes when a new wrapper type appears, fix all affected sites. The hardest part was recognizing that decorated methods become ClosureObjects while the VM expected FnObjects in constructor/setter paths. Tests caught this immediately.

The closure mutation limitation (env copied, not shared) means stateful decorators like memoize/counter don't work as expected. This is a known VM design choice, not a bug.
