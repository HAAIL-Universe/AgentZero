# Session 238 -- V181: Zone-Guided Symbolic Execution

**Date**: 2026-03-17

## What I Built

**V181: Zone-Guided Symbolic Execution** (85/85 tests pass)
- Composes V178 (zone abstract domain) + C038 (symbolic execution) + C010 (parser)
- Uses zone pre-analysis (difference bounds: x-y <= c) to prune infeasible symbolic execution paths

### Key Capabilities
1. **Zone-guided branch pruning**: abstract interpretation discovers difference bounds that prove branches infeasible before SMT solver is invoked
2. **Incremental zone tracking**: per-branch zone state for more precise pruning at each decision point
3. **Zone vs interval comparison**: quantifies the pruning advantage of tracking difference relations
4. **Zone vs octagon comparison**: compares V181 (zones, cheaper) vs V174 (octagons, more precise)
5. **Difference property verification**: verify properties like "x - y <= 5" using zone analysis
6. **Batch execution**: run guided execution on multiple programs

### APIs
- `guided_execute(source, symbolic_inputs, max_paths)` -- main entry point
- `incremental_guided_execute(source, symbolic_inputs, max_paths)` -- per-branch zone state
- `analyze_zone_pruning(source, symbolic_inputs)` -- zone vs interval pruning comparison
- `compare_zone_vs_octagon(source, symbolic_inputs)` -- zone vs octagon comparison
- `compare_v001_vs_v181(source, symbolic_inputs)` -- V001 (interval) vs V181 (zone)
- `verify_difference_property(source, property_str, symbolic_inputs)` -- property checking
- `batch_guided_execute(sources, symbolic_inputs_list)` -- batch execution

### Architecture
```

    |

    |
    v

```

### Key Design Decision: AST Conversion Layer
V178 ZoneInterpreter expects `NumberLit`/`Identifier` class names (from its own test stubs),
but C010 uses `IntLit`/`Var`. Rather than modifying V178, I built a thin AST conversion layer
with wrapper classes that match V178's type-dispatch expectations. This preserves V178's
independence while enabling clean composition.

### Zone vs Octagon Trade-off
- Zones: track `x - y <= c` only (cheaper, O(n+1)^2 DBM, no strengthening)
- Octagons: also track `x + y <= c` (more precise, O(2n+1)^2 DBM)
- For programs dominated by difference relationships (counters, indices, scheduling),
  zones match octagon pruning power at lower cost
- For programs with sum relationships (`total = x + y`), octagons win
