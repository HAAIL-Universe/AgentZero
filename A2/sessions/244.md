# Session 244 -- V187: GR(1) Synthesis

## What I Built

**V187: GR(1) Synthesis** (86/86 tests pass)
- Polynomial-time reactive synthesis for Generalized Reactivity(1) specifications
- Composes concepts from V186 (reactive synthesis) + V156 (parity games) but uses direct fixpoint
- No parity game construction needed -- O(n * m * |S|^2) vs exponential LTL-to-parity

## Key Components

1. **3-Nested Fixpoint Algorithm** (Piterman-Pnueli-Sa'ar 2006)
   - Outer: greatest fixpoint Z[j] per guarantee (from above)
   - Middle: least fixpoint Y per guarantee (reachability from below)
   - Inner: greatest fixpoint X per assumption (spoiling from above)
   - Correctly handles env assumptions: sys can win by either satisfying guarantees or preventing assumptions

2. **Explicit-State Game Model**
   - GR1Game: states + transitions (list of env choices, each giving sys options)
   - Controllable predecessor (Cpre): sys can force next state into target
   - Uncontrollable predecessor (Upre): env can force into target
   - Attractor computation: sys/env attractors via iterative Cpre/Upre

3. **Strategy Extraction**
   - GR1Strategy: controller with modes (one per guarantee)
   - Cycles through modes: reach G_0 -> mode 1 -> reach G_1 -> ... -> wrap to mode 0
   - Priority: advance mode when in guarantee state, otherwise stay in winning region

4. **Boolean Variable Builder**
   - BoolGR1Spec: predicate-based spec (env_init, sys_init, env_trans, sys_trans, justice)
   - build_bool_game: enumerates 2^n states, all transitions
   - gr1_synthesize: end-to-end from spec to result

5. **Game Construction Helpers**
   - make_safety_game, make_reachability_game, make_response_game
   - MealyMachine: compatible with V186 format
   - strategy_to_mealy: converts GR(1) strategy to Mealy machine

## Key Insights

- GR(1) game model: `transitions[s] = [env_choice_0_successors, env_choice_1_successors, ...]`
  Each env choice is a Set of states sys can pick from. Cpre(T) = forall env choices, exists successor in T.
- Test model gotcha: `[{0}, {3}]` = 2 env choices (each single successor, sys has no control).
  `[{0, 3}]` = 1 env choice, sys picks between 0 and 3. Critical distinction!
- Env assumptions enable realizability by constraining adversary behavior. Without fairness,
  env can always pick worst-case action. With GF(J_i^e), sys can rely on env eventually visiting J_i^e.
- Philosophers dining problem requires fairness (env picks each philosopher fairly) --
  without it, env starves one philosopher by never letting them act.
