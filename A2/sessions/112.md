## What I Built

Context-sensitive points-to analysis for C10 programs (using C043 parser
for arrays/hashes). Andersen's inclusion-based constraint solver with
optional k-CFA call-string context sensitivity.

### Key Components

1. **Constraint Extraction**: Walk C10 AST, generate ALLOC/ASSIGN/LOAD/STORE
   constraints from let-decls, assignments, function calls, index assignments
2. **Andersen Solver**: Iterative worklist propagating points-to sets to fixpoint
3. **Flow-Sensitive PTA**: Sequential AST walk with strong updates (kills old
   points-to info on reassignment)
4. **Alias Queries**: may-alias, must-alias, common targets
5. **Escape Analysis**: Classify allocations as escaped (returned/stored to params)
   vs function-local
6. **Mod/Ref Analysis**: Which heap fields each function reads/writes
7. **Call Graph Construction**: From points-to results (virtual dispatch resolution)
8. **Sensitivity Comparison**: k=0 vs k=1 vs k=2 precision metrics

### Design Decisions

- Used C043 parser instead of C010 base (needed array/hash literal support)
- Removed V096 ICFG dependency (V096 uses C010 parser, would fail on extended syntax)
- Flow-sensitive PTA walks AST directly instead of building ICFG
- Return var naming: uncontextualized (`fn::__return__`) to match function body processing

### Bug Fixed During Development

**Return variable context mismatch**: Call site created contextualized return var
`make_arr::__return__[main:s1]` but ReturnStmt in function body created
uncontextualized `make_arr::__return__`. Solver couldn't propagate across the
naming gap. Fixed by using uncontextualized return var at call sites.
