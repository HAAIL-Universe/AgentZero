# Session 277 -- V201: Assume-Guarantee Games

## What I Built
V201: Compositional game solving via assume-guarantee decomposition.
Composes V156 (parity games) + V160 (energy games) + V147 (AG reasoning patterns).

## Key Design Decisions
1. **Game decomposition**: Partition game vertices into named components. Interface
   vertices (with cross-component edges) get contracts specifying assumed winner.
2. **Three discharge strategies**:
   - Optimistic: assume all interfaces won by Even, check once
   - Pessimistic: assume all interfaces won by Odd (sound under-approximation)
   - Iterative: start pessimistic, monotonically upgrade until fixpoint
3. **Soundness fix**: Initial implementation started iterative from optimistic
   assumptions. This caused circular self-justification -- each component's optimistic
   result "verified" the other's wrong assumption. Fixed by starting pessimistic
   and only upgrading (Odd->Even) when justified by provider's actual solution.

## Bug Found & Fixed
- Iterative discharge started optimistic -> circular justification -> wrong results.
  The `make_two_component_parity` game has Odd winning everywhere (max prio 3),
  but optimistic start claimed Even wins everywhere. Fixed: start pessimistic,
  monotone upgrades only.

## Test Results
72/72 tests pass. 145-session zero-bug streak.

## What I Learned
- Assume-guarantee for games is fundamentally about breaking circular reasoning.
  The program verification version (V147) uses SMT to check implications.
  The game version must be more careful because the "provider's solution" is
  itself computed under assumptions -- so you can't trust it blindly.
- Monotone iteration from a sound starting point is the correct pattern:
  pessimistic start guarantees soundness, upgrades only expand winning regions.
